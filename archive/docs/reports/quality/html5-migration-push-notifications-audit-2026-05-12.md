# HTML5 Migration Push Notifications Audit

Date: 2026-05-12
Change: `implement-html5-migration-experience-optimization`
Task focus: `2.5.1` through `2.5.5`
Scope: Desktop-only, repo-local audit only

## Decision

`2.5.1` through `2.5.5` remain open.

This batch records the current push-notification surface. The repo has notification preferences, email/WebSocket notification capabilities, front-end subscription wrapper names, and service-worker push/click handlers. It does not yet provide an end-to-end browser Push API capability.

## Evidence Checked

Commands:

```bash
rg -n "Notification\\.requestPermission|PushManager|serviceWorkerRegistration\\.pushManager|subscribeToNotifications|unsubscribeFromNotifications|/subscribe|/unsubscribe|notification preferences|push notification|Web Push|notificationclick|push" web/frontend/src web/frontend/public web/frontend/tests web/backend/app/api/notification.py web/frontend/package.json
sed -n '1,240p' web/backend/app/api/notification.py
sed -n '1,220p' web/frontend/src/api/user.ts
sed -n '400,450p' web/frontend/public/sw.js
sed -n '1,220p' web/frontend/src/views/system/Settings.vue
sed -n '1,220p' web/frontend/src/views/artdeco-pages/settings/NotificationSettings.vue
```

Observed repo facts:

- `web/backend/app/api/notification.py` exposes notification preferences and email/WebSocket-style notification capabilities.
- `web/frontend/src/api/user.ts` includes client-side notification methods and subscription-oriented wrapper names.
- `web/frontend/public/sw.js` contains `push` and `notificationclick` listeners.
- `web/frontend/src/views/system/Settings.vue` is the current active settings route surface and mentions the `/api/notification/preferences` contract.
- `web/frontend/src/views/artdeco-pages/settings/NotificationSettings.vue` contains a richer notification-preference UI fragment, but it is not the active routed settings truth inspected here.

## Gap Summary

The repo has notification-related pieces, but not a complete Web Push implementation.

No active `Notification.requestPermission` flow was found. No `PushManager` subscription creation flow was found. No verified backend Web Push subscription management route was confirmed in the current notification API. The service-worker `push` handler can display a received payload, but the repo does not yet show how a user grants permission, creates a browser subscription, stores it, and receives a market alert generated by the system.

The active settings page does not expose a notification-preferences form. The richer ArtDeco notification settings fragment remains separate from the current canonical settings route.

## Task Disposition

Keep all `2.5.x` items unchecked until a later approved batch defines and verifies an end-to-end Desktop Web Push path.

Minimum future evidence should include:

- Browser notification permission UI and state handling.
- `PushManager` subscription and unsubscribe flow.
- Backend subscription storage/management routes.
- Integration from market/risk alerts to Web Push payload creation.
- Active settings route ownership for notification preferences.
