### 3. 健康端点数字仍有口径风险

事实基线在 `backend-audit-phase3-fact-baseline-2026-05-17.md:178` 写“碎片健康端点 11 个”，但本次对 `web/backend/app` 的 route decorator 快速扫描发现 21 个包含 health 的 route decorators。这个 21 包括 canonical、系统级、fragment、模型示例等不同类型，不能直接替代“碎片健康端点”数字；但它说明当前文档还缺少可复现分类规则。

文档在 `backend-audit-phase3-fact-baseline-2026-05-17.md:192` 又写“announcement, wencai, signal_monitoring 等域另有 health 端点”，这会让“11 个”看起来像是不完整清单，而不是冻结基线。

建议:

- 将健康端点拆成三类：
  - canonical health endpoints。
  - active fragmented health endpoints。
  - excluded health-like routes，例如 examples、detailed subresource health、non-router model examples。
- 每一类都给出生成命令和排除规则。
- P3-C7 的标题不要写“11+”这种浮动数字；应引用 route table baseline 的具体分类 ID。

### 4. P3-B2 `Core validation ×3` 与事实基线不一致

事实基线列出 5 个校验相关文件：

- `app/core/validation.py`
- `app/core/validators.py`
- `app/core/validation_messages.py`
- `app/core/strategy_validator.py`
- `app/core/data_validator.py`

见 `backend-audit-phase3-fact-baseline-2026-05-17.md:58-63`。

但修订计划在 `backend-audit-phase3-plan-revised-2026-05-17.md:82` 仍写 `Core validation ×3 合并`，且标为 `not required`、`Blocked by none`、低风险。

问题:

- 标题与事实基线冲突，执行者会不知道只合并 3 个还是先分类 5 个。
- validation canonical 合并属于 canonical module ownership 变更，至少需要 decision record；是否 OpenSpec 豁免需要写明理由。
- 如果 `strategy_validator.py` 和 `data_validator.py` 是业务 validator，直接纳入 Core validation 合并可能扩大影响面。

建议:

- 将 P3-B2 改为 `Core validation files classification and canonical decision`。
- 先输出分类表：generic validation、message catalog、strategy-specific validator、data-specific validator。
- 若只处理 3 个 generic 文件，标题和 Scope 必须明确排除 `strategy_validator.py`、`data_validator.py`。
- OpenSpec 字段建议从 `not required` 改为 `maybe`，直到 decision record 说明豁免理由。

### 5. P3-C dependency graph 仍有反向或无关依赖

`backend-audit-phase3-plan-revised-2026-05-17.md:51` 将 P3-C8 `Core 子目录拆分` 阻塞在 P3-C3 `Core singleton→Depends` 之后；`backend-audit-phase3-plan-revised-2026-05-17.md:99` 的表格也重复了这个依赖。

但从架构顺序看，Core 子目录拆分通常应先稳定 module ownership 和 import compatibility，再做 singleton→Depends 迁移。否则 DI 迁移会在即将移动的 database/security/socketio 文件上发生，增加重复改动和冲突风险。上一版 Phase 2 深化机会中也暗示 D-10 依赖 D-5，而不是相反。

另一个问题是 `backend-audit-phase3-plan-revised-2026-05-17.md:53` / `:101` 将 P3-C10 `Schema 双目录合并` 阻塞在 P3-C9 `Core cache` 归并之后。schema 双目录与 Core cache 迁移没有明显结构依赖，这会制造不必要的串行等待。

建议:

- 将 P3-C8 作为 P3-C3 的前置或并行前置，而不是后置。
- P3-C9 可以依赖 P3-C8，因为 cache 归并属于 Core 子目录迁移的一部分。
- P3-C10 应独立依赖自己的 schema decision / OpenSpec proposal，不应依赖 P3-C9。

### 6. P3-B6 七域 flat→package 收口仍不应整体标为 `not required`

`backend-audit-phase3-plan-revised-2026-05-17.md:86` 把七个域的 flat→package 收口合并为一个 P3-B6，标记 `not required`，风险 `低-中`。

这仍然偏粗。七个域虽然被归为低复杂度，但每个域都可能涉及 route prefix、import 引用、OpenAPI path、测试引用和 compatibility shim。根据 `architecture/STANDARDS.md` 的迁移收口规则，迁移任务应自带收口条件、兼容层退役条件和审计口径。

建议:

- P3-B6 只能作为 parent issue 或 epic。
- 每个域拆成一个 independently-grabbable issue。
- 每个子 issue 都必须包含：
  - 当前 flat route 列表。
  - package route 覆盖情况。
  - expected OpenAPI diff。
  - import/test references。
  - flat 文件删除或 shim 保留的退役条件。
- OpenSpec 字段在 route table baseline 产出前应为 `maybe`，不是统一 `not required`。

## High Priority Findings

### 7. Announcement 消费者断言需要证据化

事实基线在 `backend-audit-phase3-fact-baseline-2026-05-17.md:144` 写“前端和测试全部使用 `/api/announcement/*`，`/api/v1/announcement/*` 几乎无消费者”。这个判断方向合理，但文档没有给出消费者扫描命令或摘要。

建议在事实基线或 P3-A1 decision issue 中补：

- `web/frontend/src` 中 announcement API 引用统计。
- `web/backend/tests` 和 `tests` 中 announcement route 引用统计。
- `/api/v1/announcement` 引用统计。
- `web/frontend/src/services/versionNegotiationPolicy.ts` 是否包含 announcement 相关策略。

P3-B1 修复双注册前，这些证据必须进入 P3-A1 decision record。

### 8. Issue template 已改善，但仍缺少本地 issue tracker 要求的字段

修订计划的模板在 `backend-audit-phase3-plan-revised-2026-05-17.md:107-146` 已经明显优于原版，但与 `docs/agents/issue-tracker.md` 的本地规则相比，还缺少或不够显式：

- `Non-goals`
- `Triage label`
- `Ready-for-agent criteria`
- `Verification evidence expected`
- `Rollback / compatibility exit condition`

建议补到模板中。尤其是 `Triage label` 很重要：未完成决策或 proposal 的 issue 不应被标成 `ready-for-agent`。

### 9. P3-A decision issues 需要定义 decision record 格式

P3-A1/P3-A2/P3-A3/P3-A4 都是决策类 issue，但修订计划只写了 output 类型，没有定义 decision record 必须包含什么。

建议增加 `Decision Record Template`：

- Context
- Current facts and baseline references
- Options considered
- Decision
- Compatibility policy
- OpenSpec required?
- Follow-up implementation issues unlocked
- Verification / rollback strategy

这样 P3-B/P3-C 的 `Blocked by` 才能被真正解除。
