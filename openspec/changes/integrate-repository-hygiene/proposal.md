# Change Proposal: integrate-repository-hygiene

> **专题方案说明**:
> 本文件用于描述某一专题能力的规格边界、变更提案或专题约束，服务于 OpenSpec 的方案管理与差异追踪。
> 它不自动等同于“当前已上线实现”或仓库共享治理规则的唯一真相源；执行时需同时核对 `architecture/STANDARDS.md`、审批状态、当前代码实现以及相关 `openspec/specs/` 正式规格。


## Why

当前仓库已经具备 **policy-driven directory governance** 基础能力，但仍存在三个结构性断点：

- `docs/FILE_CLEANUP_TASK.md` 的清理任务基线已经落后于当前仓库现状
- P1/P2 中的日志轮转、自动清理、文件大小监控、文档整理尚未收敛到统一治理模型
- 仓库里已经存在多套相似脚本与目标目录约定，继续并行追加会进一步放大维护成本

当前最需要的不是再新增一组“清理动作”，而是把 **目录治理、运行时卫生、文档生命周期管理** 统一到一个后续变更中，使策略、目录层级、脚本入口与治理报表保持一致。

## What Changes

本变更将把现有目录治理扩展为 **repository hygiene governance**：

- 将 P1/P2 任务与现有目录治理方案整合为同一条治理主线
- 明确并落地 canonical lifecycle directories：`docs/`、`reports/`、`archive/`、`var/`
- 为日志轮转、自动清理、文件大小监控提供统一入口与 dry-run 机制
- 收敛现有重复的清理/监控脚本，减少脚本分叉
- 将文档治理从“按文件搬运”升级为“按生命周期落位”
- 用 OpenSpec 约束后续实现顺序：先定规则与目标目录，再逐步迁移存量债务

## Scope

### In Scope

- 更新目录治理与文件组织规格
- 设计统一的仓库卫生治理模型
- 规划 `auto_cleanup`、`monitor_file_size`、日志轮转、归档目录结构的统一入口
- 规划 `docs/`、`reports/`、`archive/` 的生命周期边界
- 制定分阶段收敛与验证策略

### Out of Scope

- 本次变更内立即完成全仓文档搬迁
- 本次变更内一次性清除全部历史 warning / tolerated debt
- 本次变更内做大规模自动移动文件的高风险操作
- 本次变更内处理与目录治理无关的前后端技术债

## Expected Impact

- 让 P1/P2 清理工作不再以零散脚本存在，而是纳入统一治理体系
- 为后续根目录 `error` 级问题收敛提供可执行的正式目标目录
- 降低运维脚本与文档结构持续分叉的风险
- 让“检查器 -> 清理器 -> 归档器 -> 报告”形成闭环

## Risks and Mitigations

- **风险：目标目录定义不一致，导致新规则互相冲突**
  - **缓解**：先更新规格与策略边界，再创建目录与脚本
- **风险：自动清理脚本误删文件**
  - **缓解**：默认提供 `--dry-run`，仅在显式确认后执行变更
- **风险：文档整理范围过大，影响正在使用的说明文档**
  - **缓解**：先做分类规则和迁移清单，再分批执行
- **风险：继续新增重复脚本**
  - **缓解**：要求新增治理入口必须复用或替换已有同类脚本
