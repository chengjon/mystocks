## Issues

- [x] **[MED]** `related_function_tree_node` missing from evidence generation specification — design:74 (evidence JSON) and tasks 5.1-5.5
      Evidence: Upstream guide section 7.1 line 271 requires `related_function_tree_node` in every evidence JSON. Grepped the entire change directory for `related_function_tree_node` — zero matches. The design ledger (design:82-112) and task list both omit this field. The field maps evidence to a specific node in the miniQMT function tree for promotion gate traceability.

- [x] **[MED]** `hash_or_size` evidence contract field not specified — design:74 (evidence JSON) and tasks 5.1-5.5
      Evidence: Upstream guide section 7.1 lines 266-269 require a `hash_or_size` object mapping raw source files to their SHA-256 and byte counts. Tasks 5.2 mentions `raw_source_artifacts` which may serve the same purpose, but the naming and structure diverge from the upstream contract. Without explicit alignment, the generated evidence may fail miniQMT's `validate_market_data_promotion_evidence.py` validator.

- [x] **[MED]** Bundle mode structure not specified against upstream bundle format — design:51 (bundle mode) and tasks 6.2
      Evidence: Upstream guide section 10 defines a specific promotion bundle with 15 named files (`promotion_requirements.json`, `mystocks_dry_run.template.evidence.json`, `bundle_manifest.json`, etc.) generated by `prepare_market_data_promotion_bundle.py`. The design mentions "bundle mode" generically and the CLI task accepts `--bundle-path`, but neither document specifies the expected bundle directory layout. Implementers may create an ad-hoc format incompatible with the upstream bundle.

## Suggestions

- Add a subsection under "Proposed Components" showing the exact `evidence.v1` JSON structure MyStocks will emit, using the upstream guide section 7.1 example as a template with MyStocks-specific values filled in. This gives implementers a concrete target and makes it easy to diff against future upstream contract changes.

- Clarify how `miniqmt_preview_status` and `miniqmt_apply_status` ledger fields get updated. The upstream guide section 9.5-9.6 shows these are miniQMT operator actions external to MyStocks. State whether MyStocks polls the miniQMT API, accepts manual input, or treats these as purely informational audit columns.

- Specify the bundle mode fixture layout explicitly (e.g., "expects a directory containing `bundle_manifest.json` and at least one artifact file matching the manifest entry"). Reference the upstream guide section 10 bundle manifest format.

- Consider adding `field_mapping_version` to the CLI output alongside hashes and row counts, since the upstream guide section 8 recommends it as an advisory field and the design already includes it in the ledger.
