## 整体评价

这是一份**经过充分压测的高质量 OpenSpec 提案**。7 轮 grilling 的决策时间线、三层契约的分层收口、TEMP_OVERRIDE 的治理细则都体现了严谨的工程思维。问题集中在 **spec ↔ design 接口签名不一致** 和 **部分 spec scenario 不可测或缺少错误路径**，都属于可修复范围。

**亮点**:
- 7 轮 grilling 时间线 + 否决项是极好的审计痕迹
- Round 2 明确否决了 A/B/C 三个反模式方案，防止后人重新提案

**问题**:

| # | 严重 | 问题 |
|---|------|------|
| D1 | 🔴 | **§5 接口签名定稿缺少 `ExtraSourceNameConflictError`**。spec.md 的 "Duplicate adapter name rejected" scenario 引用了该类，但 design 代码草稿只定义了 `ExtraSourceCategoryConflictError`。必须补齐 |
| D2 | 🟡 | **`fallback_triggered: bool = False` 字段矛盾**。注释写「extra source 单源,永远 False」。若永远不变 → 应删除（YAGNI）；若为未来预留 → 注释应改为 `# reserved for future multi-source extra adapters` |
| D3 | 🟢 | `OPENSTOCK_STATIC_CATEGORIES` 硬编码自 2026-07-02 快照，文档说「未来可用 OpenStock `/sources/categories` endpoint 替代」，但没有写 drift 检测机制。建议加一行注释：「每季度人工比对一次 `DATA_CAPABILITY_SCOPE.md`，或 OpenStock 落地 endpoint 后改动态加载」 |
| D4 | 🟢 | TEMP_OVERRIDE 90 天硬上限到期后的行为未定义——是强制下架（功能断裂）还是自动转为常规 ExtraSource？建议补一行：「到期后若 OpenStock 仍未覆盖，需人工重评估：转常规 ExtraSource 或延期（需审批）」 |

### tasks.md ✅ 可执行

**亮点**:
- 5 Phase 分层合理，Phase 1 可独立合并
- Non-Blocking Follow-ups 明确列出 C1/C2/C7 后续提案，不混入本提案
- Phase 4 仅做 issue tracking，无代码改动——边界清晰

**问题**:

| # | 严重 | 问题 |
|---|------|------|
| T1 | 🟢 | Phase 5.3「运行项目原生检查(**任一通过即可**)」疑似笔误。pytest / ruff / black 不是互替检查——应改为「**全部通过**」 |
| T2 | 🟡 | Phase 3.1 CI 脚本 registry 来源写「扫描所有 `register_extra_source` 调用点(或 registry 配置文件,实施时决定)」。design §5 已定义 `_registered` dict，最干净的实现是在 lifespan 后 dump JSON snapshot → CI 读 JSON。建议去掉「实施时决定」，直接指定方案 |
| T3 | 🟡 | Phase 1.3 测试清单缺「TEMP_OVERRIDE（带 expires_on）仍通过静态注册」这条 case。design 明确说 expires_on 不影响静态校验，但测试不覆盖会导致实施者误判 |
| T4 | 🟢 | Phase 2.1 `extra_source_fallback_count` 注释写「暂不接 Prometheus」。一个不接任何消费者的 counter 是死代码——要么至少接 `logging.info`，要么删掉 |
| T5 | 🟢 | Phase 4 依赖外部操作（双仓开 issue），提案无法自证完成。建议加 gate：「Phase 4 完成标志 = 两个 issue URL 填入 tasks.md 本条目」 |

### spec.md 🟡 缺错误路径 + 1 个不可测 scenario

**亮点**:
- 5 Requirement × 11 Scenario，Gherkin 格式规范
- Happy path + error path + edge case 覆盖基本齐全
- 每个 Scenario 有明确的 Given/When/Then/So that

**问题**:

| # | 严重 | 问题 |
|---|------|------|
| S1 | 🔴 | **"Duplicate adapter name rejected" scenario 引用 `ExtraSourceNameConflictError`，该类在 design.md §5 接口签名中不存在**（同 D1） |
| S2 | 🟡 | **"Static category always uses OpenStock" scenario 不可测**——Gherkin body 自己写「this is impossible because registration would have failed; this scenario is documentation of the invariant」。这不是可测试的 scenario，应该删除或移到 design.md 作为设计注释 |
| S3 | 🟡 | **缺 ExtraSource adapter `fetch()` 异常处理 scenario**。当 adapter 的 `fetch(params)` 抛出异常时，路由层应捕获并包装为 `DATA_GATEWAY_UNAVAILABLE` 还是 500 炸给调用方？spec 完全没有定义。这直接影响 tasks.md Phase 2.1 的 `HybridDataSource` fallback 链实现 |
| S4 | 🟡 | **生产模式 `UNSUPPORTED_CATEGORY` 行为被埋在 Mock scenario 括号里**——「falls back to MockDataSource (test mode) or returns UNSUPPORTED_CATEGORY (production mode)」。生产行为应有独立 scenario |
| S5 | 🟢 | **"Adapter implementing the Protocol" scenario 中「And no import of akshare is required」不可测试**——`isinstance` 检查不关心 import 路径。这是实现建议，应删除或改为注释 |
| S6 | 🟢 | **"Category overlaps" scenario 中「exits with a non-zero status code」措辞不精确**——FastAPI lifespan 中未捕获的异常表现为 Python traceback + 进程崩溃，不是干净的 `exit(1)`。建议改为「And the FastAPI application fails to start with ExtraSourceCategoryConflictError in the startup logs」 |
