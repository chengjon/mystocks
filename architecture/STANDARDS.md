# MyStocks ArtDeco 工程红线与架构准则 (V3.1)

本文档定义了开发过程中必须严格遵守的架构约束和开发流程。所有代码提交必须符合这些规则。

## 零、统一治理与审批门禁 (Single Source of Truth)

### 1. 方案先行准则 (Proposal-First Rule)
*   **严禁自主实施**：任何涉及菜单结构、UI/UX 风格、核心架构的变更，必须先提交《技术方案/设计草案》。
*   **审批后执行**：只有在用户明确回复“批准”、“同意”或“执行”后，方可启动代码修改。
*   **违规回滚**：任何未经审批的自主变更均被视为无效交付，必须按需无条件回滚。

### 2. 共享规则归一化
*   **唯一来源**：以下共享规则以本文件为唯一事实来源：`方案先行准则`、`六步走战略`、`Docker 一等公民原则`、`迁移收口与技术债治理规则`。
*   **引用优先**：`CLAUDE.md`、`AGENTS.md`、`.gemini/GEMINI.md` 应以引用本文件为主，避免复制粘贴同一规则正文。
*   **更新顺序**：若规则调整，先更新本文件，再同步各入口文档的引用说明。

## 一、 推荐开发流程：六步走战略

### 1. 契约先行 (Contract First) —— 避免“接口分裂”
*   **原则**：严禁在未定义契约前编写业务逻辑。
*   **契约定位**：OpenAPI 是接口契约，不是事后补写的说明文档。前端、后端、测试、第三方集成必须以同一份契约为准。
*   **单源可信**：当前仓库以 `FastAPI 路由 + Pydantic Schema + 导出的 /openapi.json` 为 API 契约唯一事实来源；Markdown 文档、前端手写类型、临时调试脚本均不得充当并行真值。
*   **流程**：先定义 OpenAPI (Swagger) 规范 -> 自动生成/导出契约 -> 运行 `generate_frontend_types.py` 生成 TS 类型。
*   **同步要求**：任何接口字段、参数、认证、错误响应、示例或语义变更，必须与 OpenAPI 契约在同一提交中同步更新；禁止“代码已变，契约未变”。
*   **可追溯性**：契约变更必须与代码一起纳入 Git，确保可比较差异、可回滚、可审计。
*   **收益**：前端永远不会拼错字段名，后端修改字段时前端编译直接报错。

### 2. 单体骨架 (Monolithic Skeleton) —— 避免“循环依赖”
*   **原则**：严禁循环依赖。
*   **分层约束**：
    *   **Core**: 工具类、配置、日志（无依赖）。
    *   **Domain**: 纯 TypeScript/Python 实体定义（无业务逻辑）。
    *   **Infra**: 数据库适配器、API Client、Redis（只依赖 Core 和 Domain）。
    *   **Application**: 业务服务逻辑。
    *   **UI/API**: Vue 组件 / FastAPI 路由（最上层）。
*   **拦截**：禁止底层模块依赖上层模块。

### 3. Mock 驱动开发 (Mock Driven) —— 避免“前端等后端”
*   **原则**：实现前后端开发解耦。
*   **要求**：使用 MSW 或后端 Mock 模式。在后端未就绪前，前端 UI/UX 必须能够通过假数据独立完成视觉验收。

### 4. 垂直切片 (Vertical Slicing) —— 避免“连不上库”
*   **原则**：做一个 Feature，通一个 Feature。
*   **流程**：一次只实现一个完整 Feature 的垂直流（DB -> Service -> API -> UI）。
*   **日志标准化**：严禁使用 `print`，必须统一使用 `from app.core.logger import logger`。

### 5. 集成与可观测性 (Observability) —— 避免“隐形白屏”
*   **原则**：系统运行状态必须对开发者透明。
*   **要求**：
    *   必须实现 RequestId 全链路追踪（注入 Header 与 Logger）。
    *   必须提供 `/health/ready` 探针（校验 DB/Redis 连通性）。
    *   前端 `App.vue` 启动时必须校验后端就绪状态，严禁在故障时显示空内容。

### 6. 自动化防护网 (Safety Net) —— 避免“修好 A 坏了 B”
*   **原则**：拒绝手动回归，依赖代码防护。
*   **要求**：每次涉及路由或 Layout 的修改必须通过 [scripts/run_e2e_pm2.sh](../scripts/run_e2e_pm2.sh) 进行全量冒烟测试。

## 二、 技术工程红线 (Technical Red Lines)

### 1. 前端开发红线
*   **路由纯净度**：[App.vue](../web/frontend/src/App.vue) 严禁硬编码业务组件，必须通过 `<router-view />` 渲染。
*   **路径语义化**：当 `MenuConfig` 的路径与路由实现不一致时，必须在 [router/index.ts](../web/frontend/src/router/index.ts) 中使用 `alias`。
*   **禁止硬编码样式**：视觉属性必须引用 [artdeco-tokens.scss](../web/frontend/src/styles/artdeco-tokens.scss) 变量。
*   **TRACE_ID 显化**：所有业务 Tab 必须在 UI 预留 Request ID 展示位。

### 2. 后端开发红线
*   **单例防御**：所有 `global` 变量必须在模块顶层显式初始化为 `None`，杜绝 `NameError`。
*   **导入安全性**：重构后必须全局搜索并清理已废弃模块的引用，防止后端陷入重启死循环。
*   **响应标准化**：所有 API 必须返回 `UnifiedResponse` 包装结构。

### 3. 环境一致性
*   **PM2/Docker 优先**：`ecosystem.config.js` 及其配套环境是“一等公民”。严禁依赖碎片化的命令行启动。

## 三、 迁移收口与技术债治理规则 (Migration Closure & Debt Governance)

### 1. 单一真相源 (Single Source of Truth)
*   **同一职责只允许一个主实现 / 主入口 / 主注册点**：适配器、数据访问层、路由入口、前端入口、兼容导出、契约定义等，不得长期并存多套“都在用”的真相源。
*   **禁止把兼容层做成平行实现**：兼容层、镜像目录、过渡模块只能作为迁移桥接层存在，不能长期升级为与主实现并行维护的第二套体系。
*   **新增实现前先定真相源**：引入新目录、新模块、新入口前，必须先声明迁移完成后的唯一保留对象。

### 2. 迁移必须自带收口条件 (Migrations Must Carry Closure)
*   **任何迁移 / 重构 / 收敛动作，在实施前必须明确**：
    *   目标真相源是谁
    *   需要保留的兼容面有哪些
    *   现有调用方 / 消费方在哪里
    *   使用什么验证命令证明迁移完成
    *   旧层何时、以什么条件下线
*   **没有退出条件的迁移禁止启动**：若无法说明旧层怎么退场，就不能引入新的平行层。
*   **架构收敛属于审批项**：涉及目录合并、路由收敛、数据访问层合并、适配器层重组、大小写目录统一、兼容层下线等事项，必须先走 `Proposal-First Rule`。

### 3. 兼容层、临时层与机械拆分约束
*   **兼容层只允许做薄封装**：root-level shim、re-export、alias route、兼容服务只允许承载最薄的一层转发，禁止继续沉淀业务逻辑。
*   **临时命名必须可追溯、可退场**：`*_new.py`、`main-*.js/ts`、`.bak`、`.backup`、`converted.archive/`、实验入口、临时目录等，只能在有明确治理任务、owner 和退出条件时短期存在。
*   **禁止机械拆分**：`part1.py`、`part2.py`、`part3.py` 一类按体积硬切的拆分，不得作为长期结构；文件拆分必须按职责、语义边界和依赖边界完成。

### 4. 清理 / 删除决策标准
*   **“未引用 / 未使用” 不等于“可删除”**：禁止仅凭静态搜索、IDE 提示、lint 告警或“当前文件内未使用”就直接删除文件、模块、组件、函数、测试、配置或导入。
*   **发起任何清理 / 删除 / 裁剪前，必须同时完成两层判定**：
    *   **代码路径判定**：确认对象是否仍被路由、菜单、注册表、动态导入、构建脚本、样式注入、副作用导入、兼容分支、特性开关、文档约定或运行时字符串映射使用。
    *   **功能树判定**：确认对象在当前项目功能树中的归属与状态，至少标记为：`有效`、`失效但兼容保留`、`实验/灰度`、`重复冗余`、`待判定` 之一。
*   **仅当“代码路径可安全移除”且“功能树状态明确为重复冗余或正式下线”时，才允许删除**。
*   **若功能仍有效、可能恢复、承担兼容职责，或状态无法明确，则默认不删除**；优先选择补注释、补文档、登记技术债、标记 `deprecated`、或在任务汇报中说明保留原因。
*   **低层级清理也必须确认无隐含职责**：对 `unused import`、未使用变量、未使用解构项等，也必须确认其不承担副作用初始化、类型约束、样式加载、注册触发、polyfill、扩展槽位等职责；无法证明时不得删除。
*   **凡提交含清理 / 删除动作，任务汇报中必须说明**：清理对象、所属功能节点、状态判定、删除依据、未删除原因（如有）。

### 5. 指标与审计口径标准
*   **审计、周报、方案、技术债报告必须明确区分**：实测值、历史基线、推断值、目标值。
*   **所有核心数字必须可追溯**：应给出统计范围、日期、命令、来源文件或口径说明；无法复现的数字不得写成“当前值”。
*   **禁止文档内部自相矛盾**：若不同章节采用不同统计口径，必须显式说明；不得在总览表与正文中给出互相冲突的结论。
*   **历史快照不能冒充当前状态**：旧覆盖率、旧 E2E 数字、旧 lint 结果，若未重新测量，必须带日期并标注为历史快照。
*   **无法形成硬指标时，应降级为观察项**：例如统计口径敏感或尚未复核的数据，只能写为观察结论，不得写成门禁指标。

### 6. 与现有治理文档的关系
*   **本节负责防止新的结构性技术债继续生成**：重点约束迁移收口、重复层、兼容层、清理判定与审计口径。
*   **技术债门禁、基线、豁免、周报模板** 继续以 `docs/standards/technical-debt-governance-charter-v1.md` 为执行细则。
