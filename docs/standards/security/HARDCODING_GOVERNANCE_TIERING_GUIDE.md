# 硬编码治理分级指南（通用版）

> 适用范围：任意软件项目（后端、前端、脚本、基础设施配置、测试代码）。  
> 目标：降低安全风险、提升可维护性、改善多环境部署一致性。  
> 本指南不依赖特定技术栈或组织流程，可直接裁剪后落地。

## 0. 文档定位与关系（重要）

- 本文是“硬编码治理”的统一补充规范，不替代项目既有运行手册、部署规范、安全基线。
- 当项目已有更细粒度 SOP 时，以项目内专项文档为准，本文用于统一分级、门禁、审查口径。
- 建议采用“双层文档”：
  - 通用层：本指南（分级模型、治理策略、门禁基线）。
  - 项目层：项目专项指南（环境初始化、轮换流程、审批链路）。

## 1. 术语与边界

### 1.1 硬编码定义
硬编码是指将本应可配置、可变或敏感的数据直接写死在代码或配置文件中。

### 1.2 不纳入“问题硬编码”的常见项
- 协议常量：如 HTTP 状态码、RFC 标准字段值。
- 数学常量：如 `PI`、`EPSILON`。
- 领域固定不变量：如算法定义中不可变参数（需有注释或文档依据）。
- 测试中明确标注的“虚拟值/假数据”（且不可误连生产）。

## 2. 风险分级模型

建议采用 `P0 ~ P4` 五级：

- `P0（阻断）`：高危敏感信息泄露或可直接导致生产事故。
- `P1（高）`：环境耦合强，跨环境易失败，或具明显安全/合规风险。
- `P2（中）`：影响扩展性与可维护性，短期可运行但长期高成本。
- `P3（低）`：可读性/一致性问题，风险较低。
- `P4（观察）`：可接受硬编码，记录即可。

## 3. 分级判定规则（推荐）

### 3.1 P0（阻断）
- 明文凭据：密码、密钥、Token、私钥、证书内容。
- 生产数据库连接串含账号密码。
- 真实第三方账号密钥出现在代码仓库历史中。
- 可直接触发横向攻击的信息（如内部网段、堡垒机地址+凭据）。

处理要求：
- 立即停止发布/合并，先替换为密钥管理方案。
- 完成密钥轮换（旧密钥作废）。

### 3.2 P1（高）
- 写死生产/预发主机、IP、端口、域名且无环境覆盖入口。
- 安全控制阈值写死（如鉴权开关、限流阈值、风控阈值）且无法动态调整。
- 跨环境配置分叉严重（同一值在多个模块重复写死）。

处理要求：
- 在最近一个迭代内完成配置化。
- 引入统一配置源（环境变量/配置中心/密钥服务）。

### 3.3 P2（中）
- 业务阈值写死（超时、重试、批次大小、分页大小等）但有统一默认配置可迁移。
- 前端 API 基地址、WS 地址在多文件散落写死。
- 同一硬编码值在单文件/多文件重复出现。

处理要求：
- 按模块批量治理，建立单一配置出口与常量层。

### 3.4 P3（低）
- 展示文案中的示例 URL、文档中的本地地址。
- 测试中 mock 用户名、非敏感假账号。

处理要求：
- 标注为示例值，避免与真实环境混淆。

### 3.5 P4（观察）
- 确认属于标准常量/不可变协议值，且已文档化说明。

处理要求：
- 保留，定期复核。

## 4. 治理优先级策略

推荐优先顺序：

1. `安全优先`：先清理 `P0` 和 `P1` 敏感项。
2. `故障优先`：再处理导致跨环境失败的连接信息。
3. `收敛优先`：最后治理重复阈值与分散配置。

## 5. 标准治理动作

### 5.1 敏感信息
- 从代码移除明文，改为：
  - 密钥管理系统（Vault/KMS/Secret Manager）；
  - 环境变量注入（最小权限）；
  - 部署平台 Secret（K8s Secret/CI Secret）。
- 伴随动作：
  - 密钥轮换；
  - 审计访问日志；
  - 清理历史提交（必要时）。

### 5.2 环境相关配置
- 提供三级配置层级（建议）：
  - 运行时环境变量（最高优先级）；
  - 环境配置文件（按 dev/test/staging/prod）；
  - 代码默认值（仅本地开发安全默认）。
- 禁止在业务代码中拼接固定主机地址。
- 前端必须避免散落硬编码：
  - API Base URL；
  - WebSocket 地址；
  - 鉴权/租户/功能开关相关环境参数。

### 5.3 业务阈值与策略参数
- 抽离至配置项并定义：
  - 合法范围（min/max）；
  - 默认值；
  - 变更审批责任人；
  - 生效方式（热更新/重启生效）。

### 5.4 重复硬编码（单文件内）
- 提取为：
  - 模块级常量；
  - 配置对象；
  - 工具函数参数默认值；
  - 枚举/字典映射。

### 5.5 前端专项（Vue `.env` 模式示例）
- 推荐变量分层：
  - `.env`：跨环境公共变量（非敏感）；
  - `.env.development` / `.env.test` / `.env.production`：环境差异；
  - CI/CD 注入敏感值，不在前端仓库存放真实密钥。
- Vue/Vite 示例（仅示意）：

```env
# .env.development
VITE_API_BASE_URL=http://localhost:8020
VITE_WS_BASE_URL=ws://localhost:8020
VITE_FEATURE_X_ENABLED=true
```

```ts
// config.ts
export const API_BASE_URL = import.meta.env.VITE_API_BASE_URL;
export const WS_BASE_URL = import.meta.env.VITE_WS_BASE_URL;
```

- 禁止模式：
  - 在组件/服务文件中直接写死 `http://localhost:8020`；
  - 在前端代码中放置任何真实 `token/secret/password`。

## 6. CI/CD 门禁建议

### 6.1 最低门禁
- 检测到 `P0`：流水线失败（阻断）。
- 新增 `P1`：默认失败，可临时豁免（需到期时间）。
- `P2/P3`：告警并计入技术债看板。

### 6.2 建议指标
- 新增高风险硬编码数（按周）。
- 存量高风险硬编码数（趋势）。
- 硬编码治理平均修复时长（MTTR）。
- 豁免项数量与超期率。

### 6.3 检测工具链推荐（可组合）

| 工具 | 主要用途 | 建议接入点 | 备注 |
|---|---|---|---|
| `detect-secrets` | 识别仓库中的高熵密钥/凭据 | pre-commit + CI | 支持 baseline，便于增量治理 |
| `gitleaks` | 扫描提交历史与当前代码泄露 | CI + 发布前 | 适合做阻断门禁 |
| `trufflehog` | 深度扫描历史与远程仓库泄露 | 安全巡检 | 对历史泄露排查有效 |
| `bandit` | Python 安全静态分析 | CI（Python项目） | 补充代码级安全问题 |

推荐最小组合：
- 提交前：`detect-secrets`（快速反馈）。
- CI 阶段：`gitleaks` + `bandit`（门禁）。
- 周期巡检：`trufflehog`（历史深扫）。

示例命令（按需）：

```bash
# detect-secrets
detect-secrets scan > .secrets.baseline
detect-secrets audit .secrets.baseline

# gitleaks
gitleaks detect --source . --verbose

# trufflehog
trufflehog filesystem .

# bandit
bandit -r src/
```

## 7. 例外（豁免）机制

允许豁免，但必须记录以下字段：
- `id`：唯一编号；
- `级别`：P0~P4；
- `文件/位置`；
- `豁免理由`；
- `责任人`；
- `创建日期`；
- `到期日期`；
- `替代控制`（如网络隔离、只读权限、访问审计）。

规则：
- `P0` 原则上不允许长期豁免。
- 任何豁免必须有到期日，超期自动升级告警。

## 8. 代码评审检查清单（通用）

- 是否新增明文凭据或连接串？
- 是否新增环境耦合地址（IP/域名/端口）？
- 同一阈值是否在多个位置重复出现？
- 配置是否可在不改代码前提下切换环境？
- 是否提供了默认值边界与异常处理？
- 是否更新了配置文档与运行手册？

## 9. 组织落地路线（建议）

### 阶段 1（1-2 周）：止血
- 建立扫描规则与 P0/P1 门禁。
- 清理明文凭据并完成轮换。

### 阶段 2（2-6 周）：收敛
- 统一配置入口，消除分散地址与端口。
- 批量处理重复阈值与连接参数。

### 阶段 3（持续）：常态化
- 每次 PR 增量扫描。
- 技术债看板周度复盘。
- 新模块默认“无硬编码敏感信息”基线。

### 阶段 4（应急）：密钥泄露处置与轮换
1. 事件分级与冻结：
   - 识别泄露范围（代码、日志、制品、聊天记录）。
   - 暂停相关发布与自动化任务。
2. 立即失效旧凭据：
   - 在密钥源头撤销旧密钥（数据库/云平台/第三方 API）。
   - 记录撤销时间与影响系统。
3. 生成并分发新凭据：
   - 新密钥最小权限、最短有效期。
   - 仅通过 Secret 管理系统/CI Secret 分发。
4. 清理仓库历史（必要时）：
   - 使用 `git filter-repo` 清理敏感字符串；
   - 强制推送并通知所有开发者重置本地历史；
   - 对 fork/镜像仓库执行同等清理。
5. 验证与复盘：
   - 运行泄露扫描工具二次确认；
   - 补充防护规则并更新门禁。

`git filter-repo` 示例（示意）：

```bash
# 1) 准备 replacements.txt
# 规则格式示例：regex:<旧值>==><新值>

# 2) 重写历史
git filter-repo --replace-text replacements.txt

# 3) 推送（需评估影响）
git push --force --all
git push --force --tags
```

注意：
- 历史重写会改变 commit hash，需提前发布迁移通知。
- 仅清理历史不等于完成修复，必须同步完成密钥轮换。

## 10. 附录：快速判定表

| 场景 | 建议级别 | 是否阻断 | 典型动作 |
|---|---|---|---|
| 明文数据库密码 | P0 | 是 | 移除+轮换+审计 |
| 写死生产 IP/域名 | P1 | 通常是 | 配置化+环境分层 |
| 写死超时/重试次数 | P2 | 否 | 提取配置+范围校验 |
| 示例文档中的 localhost | P3 | 否 | 标注示例用途 |
| 标准协议常量 | P4 | 否 | 保留并文档化 |

## 11. 推荐交付物模板（可复用）

### 11.1 治理台账字段
- `rule_id`, `severity`, `file`, `line`, `snippet`, `owner`, `status`, `created_at`, `due_date`, `closed_at`

### 11.2 任务拆分模板
- 任务名：`[P1] 配置化 API Base URL`
- 输入：受影响文件列表
- 输出：统一配置入口 + 回归验证
- 验收：增量扫描无新增高风险项，相关测试通过

## 12. Vault 集成最小示例（通用）

### 12.1 启动时注入（推荐）

```bash
export DB_PASSWORD="$(vault kv get -field=password secret/prod/database)"
export API_TOKEN="$(vault kv get -field=token secret/prod/thirdparty)"
```

应用仅从环境变量读取，不在代码中写默认明文值。

### 12.2 应用内读取（示意）

```python
import hvac

client = hvac.Client(url="https://vault.example.com")
client.auth.approle.login(role_id="ROLE_ID", secret_id="SECRET_ID")
secret = client.secrets.kv.v2.read_secret_version(path="prod/database")
db_password = secret["data"]["data"]["password"]
```

建议：
- 优先使用短期令牌与动态凭据；
- 设置访问审计与最小权限策略；
- 禁止把 `ROLE_ID/SECRET_ID` 再次硬编码到仓库。
