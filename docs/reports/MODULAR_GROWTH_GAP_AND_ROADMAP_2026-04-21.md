# MyStocks Modular Growth Gap & Roadmap

> **参考说明**
> 本文是对 MyStocks “模块化生长”治理理念的严格差距审视与落地路线建议，不是仓库共享规则的唯一事实来源。
> 涉及治理规则、审批门禁与执行约束时，请优先遵循 [`architecture/STANDARDS.md`](/opt/claude/mystocks_spec/architecture/STANDARDS.md)。

> **结论摘要**
> MyStocks 已完整采纳六步走治理理念，并在可观测性、自动化防护、契约治理上实现高度落地。
> 但由于存量规模与历史结构，不适合一次性重构为理想纯净形态。
> 当前最优路径是：用门禁守住增量合规，用分批迁移收敛存量债务，让架构在不中断业务的前提下持续向终态进化。

---

## 1. 更适合本项目的策略

更适合 MyStocks 的不是“全仓一次性纯净化”，而是：

- 增量强约束
- 存量分域收敛
- 兼容层带退场条件
- 门禁优先于重构

原因很直接：

- 仓库体量大，历史结构复杂
- 现有能力已在运行，不能整体回炉
- 风险主要来自漂移和回归，不是缺少架构口号
- 最需要的是“新改动不再制造新债”

---

## 2. 六步走差距清单

### 2.1 契约先行

现状：

- 已将 `FastAPI route + Pydantic schema + /openapi.json` 设为 API 单一事实源
- 已有 OpenAPI 契约治理标准
- 已有前端类型生成脚本
- 已有契约验证 CI

差距：

- 不是纯 `spec-first`
- 后端未统一采用 spec 生成存根
- 前端生成链路仍是混合式
- PR 层面的契约先行纪律还不够硬

建议：

- 保留 `code-first`
- 强化 `openapi.json -> types -> CI gate`
- 所有 API PR 增加契约变更摘要

### 2.2 单体骨架

现状：

- 分层意图明确
- 前端主链路已呈现 `view -> composable -> service/api`

差距：

- 缺少依赖方向门禁
- 存量兼容层仍多
- 历史目录边界不够统一

建议：

- 先对关键目录加依赖检查
- 只约束新增依赖与高风险边界

### 2.3 Mock 驱动

现状：

- 已有 MSW
- 已有 mock 测试
- 已有 mock 开发脚本

差距：

- 仍偏测试基础设施
- 还不是默认开发工作流
- mock 覆盖面不足

建议：

- 采用“双模式开发”
- 新页面与契约未稳定页面优先 mock
- 核心交易页优先真实后端

### 2.4 垂直切片

现状：

- 已明确要求 feature 级打通
- 业务域切分已成主线

差距：

- 存量不是按垂直切片自然生长
- `print` 等历史风格残留
- 统一验收面板不足

建议：

- 只对新需求和重构域强制垂直切片
- 清理主路径日志风格

### 2.5 可观测性

现状：

- `health/ready` 已落地
- 前端启动检查已落地
- Request ID 已贯通

差距：

- 诊断输出分散
- 回退原因表达不统一

建议：

- 统一 readiness / request_id / process_time 输出口径
- 将失败页、状态页、日志字段对齐

### 2.6 自动化防护网

现状：

- 契约门禁、前端测试、E2E、可视化门禁已存在

差距：

- 门禁偏多偏散
- 失败反馈不够统一

建议：

- 统一门禁输出摘要
- 减少重复检查

---

## 3. PM2 / Docker 定位

现状：

- 已有 PM2 配置
- 已有 Docker Compose

差距：

- 不是唯一开发入口
- 仍有碎片化命令

建议：

- 保留 PM2 与 Docker 双主入口
- 明确：
  - 本地增量开发优先 PM2
  - CI / 集成验证优先 Docker

---

## 4. 优先级建议

### P0

- 加依赖方向门禁
- 强化 API 契约变更摘要
- 收敛主入口脚本

### P1

- 收敛前端类型生成链路
- 扩大 MSW 使用范围
- 清理主路径 `print`

### P2

- 整合质量门禁摘要
- 建立 feature 模板
- 推进兼容层退场

---

## 5. 结论

MyStocks 不适合一次性重构成理想纯净形态。

更适合的路径是：

- 用门禁守住增量
- 用分批迁移收敛存量
- 用统一口径减少漂移
- 用退出条件管理兼容层

这不是折中，而是对现有仓库规模和风险结构的最优解。

## 6. 90 天执行路线图

### 阶段 1：先把增量门禁变硬

目标：

- 新改动不再继续放大结构性技术债
- 将“六步走”从规则提升为默认执行轨道

建议动作：

- 为前端关键目录增加依赖方向检查
- 为后端关键边界增加导入方向检查
- 为 API 改动增加 PR 模板中的契约变更摘要
- 为 PM2 / Docker 增加统一 repo-root 启动脚本

阶段验收：

- 新增 PR 能识别越层依赖
- API 变更 PR 必须能说明契约影响范围
- 开发入口从碎片化命令收敛为少数官方入口

### 阶段 2：选择 1 到 2 个业务域做样板收口

目标：

- 用真实业务域验证“增量强约束 + 存量分批收敛”是否可执行
- 把抽象理念变成可复制模板

建议优先域：

- `system`
- `strategy`

选择原因：

- `system` 域适合做 readiness、状态页、Request ID、主链路治理样板
- `strategy` 域兼具前端页面、后端服务、契约与测试，适合做 feature 全链路样板

建议动作：

- 统一目录边界与调用链
- 增补 mock / fallback 验收路径
- 补 feature 模板与测试清单
- 明确兼容层保留与退出条件

阶段验收：

- 至少 1 个业务域可以稳定按新模板开发
- 至少 1 个兼容层完成退场或登记退场条件
- 至少 1 套样板文档可供后续域复用

### 阶段 3：把局部样板扩展成主线制度

目标：

- 让新的开发默认落在统一轨道里
- 将分散的门禁收敛为可理解的质量体系
- 让 ArtDeco 前端体系从“可用组件集合”继续收敛为“可规范化演进的平台层”

建议动作：

- 汇总契约层、运行层、体验层门禁摘要
- 将 feature 模板推广到更多业务域
- 对高频路径的 `print` 和历史调试输出进行收口
- 建立兼容层治理看板
- 完善 `artdeco-tokens.scss`，减少视觉 token 漂移
- 统一 `Base / Domain` 组件结构，降低组件层级混用
- 构建自有 ArtDeco CLI 脚手架，固化新页面 / 新组件初始化方式
- 基于现有组件做视觉收敛与测试加固，减少 UI 回归

阶段验收：

- 团队可以清楚区分新增问题与历史债务
- 质量门禁输出可被单页摘要理解
- 主业务域开始按统一模板推进
- ArtDeco 体系具备可复制的组件生成与验收路径

---

## 7. 每阶段的验收标准

### 契约层

- API 改动必须生成新的 `openapi.json` 导出结果
- 前端类型生成结果必须可验证
- breaking change 必须被显式识别

### 结构层

- 新增依赖不得跨越既定边界
- 新增兼容层必须声明退出条件
- 新模块不得继续扩大重复真相源

### Mock / 验收层

- 新页面或新链路在后端未完成时应具备最小可验收路径
- mock 只服务于开发解耦，不得变成长期真实行为替身

### 运行层

- readiness 探针必须可用
- Request ID 必须能贯通主链路
- 失败状态必须向前端显化

### 自动化层

- 至少有一条自动验证链路覆盖主流程
- 相关改动必须落入对应门禁范围
- 门禁结果必须能说明失败原因，而不是只给出失败状态

---

## 8. 现在不应该做的事情

以下动作当前不适合作为主线：

- 全仓一次性改成纯 `spec-first`
- 为追求架构整洁而大规模目录重写
- 把所有页面强行迁到 mock-only 开发模式
- 立即要求所有老模块满足严格分层约束
- 无限叠加新的 CI，而不先收敛已有门禁

原因：

- 成本高
- 风险大
- 容易打断现有业务推进
- 会让治理本身成为新的复杂度来源

---

## 9. 推荐的首批治理抓手

如果只选 5 个最值得先做的抓手，建议是：

1. 依赖方向门禁
2. API 契约变更摘要
3. PM2 / Docker 主入口收敛
4. `system` 域样板收口
5. `strategy` 域样板收口

这 5 项的共同特点：

- 能直接减少增量债务
- 不要求推翻现有结构
- 容易形成组织级默认做法
- 能给后续收敛提供模板

---

## 10. ArtDeco 未来规范化路线

除了通用治理主线，前端体系还应沿着 ArtDeco 专项路线继续收敛。建议将以下四项视为中期规范化重点，而不是零散视觉优化任务。

### 10.1 完善 `artdeco-tokens.scss`

目标：

- 将视觉设计变量继续收敛到统一 token 层
- 减少组件、页面、样式文件内的重复字面量
- 提高主题、密度、状态色和金融终端风格的一致性

建议：

- 优先补齐高频颜色、边框、阴影、间距、层级、状态语义 token
- 逐步减少页面级样式中的直接值
- 将 token 变更纳入前端治理门禁与样式审计

### 10.2 统一 Base / Domain 组件结构

目标：

- 明确基础原子组件与业务域组件边界
- 减少“同名不同责”与“跨层堆叠”的组件组织问题

建议：

- `Base` 组件只承载通用表现与最小交互能力
- `Domain` 组件承载业务语义、组合规则和领域状态
- 禁止将业务逻辑继续沉淀到 Base 层
- 为组件目录、命名和依赖方向增加明确约束

### 10.3 构建自有 ArtDeco CLI 脚手架

目标：

- 把新页面、新组件、新样板的初始化从手工复制粘贴，升级为标准化生成
- 把结构规则、样式入口、测试骨架、Request ID 显化位等默认固化

建议：

- CLI 至少支持生成：
  - Base 组件
  - Domain 组件
  - ArtDeco 页面骨架
  - 配套样式文件
  - 最小单测 / 视觉 / 交互测试模板
- CLI 生成结果默认符合当前目录结构与治理规范

### 10.4 基于现有组件做视觉收敛与测试加固

目标：

- 先收敛已有活跃组件，而不是继续无边界扩展新组件
- 让视觉一致性和测试覆盖同步提升

建议：

- 先盘点高复用组件，再按使用频率做批次治理
- 优先为高频组件补充视觉快照、交互测试和样式门禁
- 将“视觉收敛”定义为平台治理动作，而不是单页面 cosmetic 调整

### ArtDeco 路线与主线治理的关系

这四项不应被视为独立于主线治理之外的 UI 优化事项，而应视为：

- “增量强约束”在前端平台层的具体表达
- “样板域收口”在设计系统层的具体表达
- “自动化防护网”在视觉与组件层的具体表达

换句话说：

ArtDeco 规范化路线不是附加任务，而是 MyStocks 前端主线治理继续向平台化演进的关键组成部分。

### ArtDeco 路线的推荐推进顺序

建议顺序如下：

1. 完善 `artdeco-tokens.scss`
2. 统一 `Base / Domain` 组件结构
3. 基于现有组件做视觉收敛与测试加固
4. 构建自有 ArtDeco CLI 脚手架

原因：

- token 是视觉与组件收敛的最底层依赖
- 组件分层需要先建立在稳定 token 语言之上
- 视觉收敛与测试加固应优先基于现有活跃组件完成
- CLI 最适合在结构和样式规范相对稳定后固化为脚手架

### ArtDeco 路线的阶段映射

#### P0

- 明确 `artdeco-tokens.scss` 的缺口范围
- 确定 `Base / Domain` 组件边界与命名规则
- 选出首批高频组件作为治理样板

#### P1

- 扩充 token，并减少高频组件中的直接字面量
- 对首批高频组件完成结构收敛
- 补足视觉快照、交互测试、样式门禁

#### P2

- 在稳定结构基础上构建 ArtDeco CLI
- 将新组件 / 新页面骨架生成纳入统一脚手架
- 让 CLI 默认输出符合现行治理规范的文件结构与测试骨架

### ArtDeco 路线的验收标准

#### Token 层

- 高复用颜色、间距、边框、阴影、层级具备统一 token
- 新增页面和组件默认优先消费 token，而不是直接写字面量

#### 组件层

- Base 组件与 Domain 组件职责边界明确
- 新增业务逻辑不再进入 Base 层
- 组件目录结构可被新成员快速理解

#### 测试层

- 高复用组件具备最小视觉或交互回归保护
- 视觉收敛与测试加固同步推进，而不是先做一端

#### 脚手架层

- CLI 生成的新页面 / 新组件默认具备统一结构
- CLI 生成产物无需人工大幅回改才能满足治理规则

---

## 11. 最终建议

对 MyStocks 来说，最优解不是追求某一次“架构大手术”。

真正应该追求的是：

- 新改动越来越难偏离治理主线
- 老问题越来越难继续扩散
- 兼容层越来越可追踪
- 门禁越来越能解释风险
- 架构越来越像“持续收敛的系统”，而不是“历史叠加的结果”

换句话说：

MyStocks 当前最需要的，不是一次性变成理想形态，而是持续具备向理想形态演化的工程能力。
