# Stage 4.3 全仓分阶段推广计划

## 目标
在试点验证通过后，将技术债治理门禁分阶段扩展到全仓，控制误伤并确保交付稳定。

## 推广原则
1. 先核心路径，后外围模块
2. 先“阻断新增”，后“治理存量”
3. 所有例外必须双签 + TTL
4. 基线更新默认只降不升

## 推广分层
### Phase A（第 1 周）
范围：
- `web/frontend/src/api/**`
- `web/frontend/src/composables/**`
- `web/backend/app/api/**`
动作：
- 启用 no-new-debt 全量阻断
- 启用 baseline 对比门禁

### Phase B（第 2-3 周）
范围：
- `web/frontend/src/views/**`
- `web/backend/app/services/**`
- `tests/**`（测试占位断言治理）
动作：
- 启用 KPI gate
- 固化周报机制

### Phase C（第 4 周起）
范围：全仓
动作：
- 全仓执行统一门禁
- 每迭代进行基线复盘
- 例外项到期自动清理

## 放量阈值（进入下一阶段条件）
- 误伤率 <= 10%
- CI 平均耗时增幅 <= 15%
- 双签例外合规率 >= 95%
- 连续 1 周无阻塞性事故

## 回滚策略
触发任一条件即暂停推广并回滚到上一阶段：
- 误伤率 > 20%
- 关键流水线阻塞 > 2 小时
- 出现 P0/P1 交付事故

## 责任分工
- Tech Lead：门禁策略 owner、双签审批
- 模块负责人：例外归口、整改推进
- QA/效能：误伤率统计、周报产出

## 交付物清单
- 每周：`reports/analysis/tech-debt-weekly-report.md`
- 每阶段：推广复盘记录（本文件追加）
- 每月：基线变更审计记录
