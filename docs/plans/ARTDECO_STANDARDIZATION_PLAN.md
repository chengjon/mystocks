# ArtDeco Standardization Plan

> **说明**
> 本文是 MyStocks 前端 ArtDeco 体系的专项标准化计划，用于把现有设计系统、组件体系、视觉治理和测试能力收敛为可持续演进的平台层。
> 仓库级共享规则仍以 [`architecture/STANDARDS.md`](/opt/claude/mystocks_spec/architecture/STANDARDS.md) 为准。

> **定位**
> 这不是一次性 UI 重做计划，也不是纯视觉优化清单。
> 它的目标是：在不中断业务开发的前提下，把 ArtDeco 从“可用组件集合”推进成“可规范、可生成、可测试、可治理”的前端平台。

---

## 1. 背景判断

ArtDeco 已经是 MyStocks 前端的重要主线，但当前仍存在以下结构性问题：

- token 体系尚未完全覆盖高频视觉语义
- Base / Domain 组件边界还不够稳定
- 新页面和新组件创建过程仍偏手工
- 高复用组件的视觉一致性与回归保护还不够强

因此，ArtDeco 的下一阶段重点不应是继续无边界扩张新页面，而应优先做标准化收敛。

---

## 2. 计划目标

本计划聚焦 4 个核心目标：

1. 完善 `artdeco-tokens.scss`
2. 统一 Base / Domain 组件结构
3. 构建自有 ArtDeco CLI 脚手架
4. 基于现有组件做视觉收敛与测试加固

它们之间不是并列无序关系，而是有明确依赖顺序的。

---

## 3. 推荐推进顺序

建议顺序：

1. 完善 `artdeco-tokens.scss`
2. 统一 Base / Domain 组件结构
3. 基于现有组件做视觉收敛与测试加固
4. 构建自有 ArtDeco CLI 脚手架

原因：

- token 是视觉与组件平台的底层语言
- 组件结构必须建立在稳定 token 与明确职责边界之上
- 现有组件应先收敛，避免把历史混乱固化进脚手架
- CLI 应在规范相对稳定后再固化

---

## 4. 路线一：完善 `artdeco-tokens.scss`

### 目标

- 让高频视觉语义尽可能回到 token 层
- 降低页面级和组件级样式里的直接字面量
- 提高颜色、边框、阴影、层级、状态表达的一致性

### 范围

- 颜色 token
- 边框与半径 token
- 阴影与层级 token
- 间距与密度 token
- 状态语义 token

### 优先级

- P0：补齐高频缺口
- P1：替换高频组件中的直接字面量
- P2：推动更多页面消费 token

### 验收标准

- 新增组件默认优先使用 token
- 高复用组件中的硬编码视觉值明显下降
- 主视觉语义在多个页面中呈现一致

---

## 5. 路线二：统一 Base / Domain 组件结构

### 目标

- 明确基础组件与业务域组件的职责边界
- 降低组件层级混用、命名混乱和跨层逻辑沉淀

### 结构原则

`Base` 组件：

- 只承载通用表现与最小交互能力
- 不沉淀业务语义
- 不耦合业务状态

`Domain` 组件：

- 承载业务语义和组合规则
- 可依赖 Base 组件
- 可面向业务域暴露更清晰的输入输出

### 禁止项

- Base 组件内继续沉淀业务逻辑
- Domain 组件反向变成通用组件库入口
- 通过复制组件而非明确分层来解决职责冲突

### 验收标准

- 新增组件能被清楚归类到 Base 或 Domain
- Base 层职责边界清晰
- 高复用业务组件不再依赖隐式样式约定

---

## 6. 路线三：基于现有组件做视觉收敛与测试加固

### 目标

- 先收敛存量高频组件，而不是继续制造新的平台分叉
- 让视觉一致性和回归保护同步提升

### 策略

- 先做组件盘点
- 按使用频率和影响面排序
- 先治理高频组件，再治理边缘组件

### 测试策略

- 为高频组件补最小视觉快照
- 为关键交互补最小行为测试
- 将样式门禁、视觉回归和组件结构治理联动

### 验收标准

- 首批高频组件具备最小视觉或交互回归保护
- 核心组件的样式表达更稳定
- “页面修一个、别处坏一个”的风险下降

---

## 7. 路线四：构建自有 ArtDeco CLI 脚手架

### 目标

- 将新页面、新组件、新样板的创建过程标准化
- 把结构规则、样式入口、测试骨架固化进生成器

### CLI 最小能力

- 生成 Base 组件
- 生成 Domain 组件
- 生成 ArtDeco 页面骨架
- 生成配套样式文件
- 生成最小测试骨架

### 设计要求

- 输出路径符合当前目录治理规则
- 默认集成 token 使用约定
- 默认包含最小可验证结构
- 不生成与现有治理冲突的样板

### 验收标准

- 生成结果可直接纳入正常开发
- 生成结果不需要大幅返工才能满足现行规范
- 新成员可以通过脚手架快速进入统一模式

---

## 8. 分阶段执行建议

### P0：标准冻结前的最小收口

- 梳理 `artdeco-tokens.scss` 缺口
- 冻结 Base / Domain 组件职责边界
- 选出首批高频组件样板

### P1：样板组件批次治理

- 补 token
- 收敛高频组件结构
- 增补视觉和交互测试

### P2：平台化固化

- 构建 ArtDeco CLI
- 将组件 / 页面样板生成标准化
- 用 CLI 承载已稳定的结构规范

---

## 9. 依赖关系

这 4 条路线的依赖关系如下：

```text
Token 稳定
  -> 组件结构稳定
    -> 现有组件视觉收敛与测试加固
      -> CLI 固化生成规则
```

如果顺序倒置，会出现两个问题：

- 脚手架提前固化了不稳定结构
- 组件治理仍在变动，导致生成器持续返工

---

## 10. 当前不建议做的事

- 直接先做 CLI 再回头修 token
- 在组件边界不清楚时大规模迁移目录
- 把视觉收敛理解为单页面 cosmetic 微调
- 只做视觉统一，不补测试保护
- 只做测试加固，不先解决组件结构混乱

---

## 11. 与主线治理的关系

ArtDeco 标准化计划不是独立于主线治理之外的前端美化任务，而是：

- “增量强约束”在前端平台层的落地
- “样板域收口”在设计系统层的落地
- “自动化防护网”在视觉和组件层的延伸

因此，这份计划应与仓库整体 Modular Growth 治理路线一起推进，而不是作为孤立的设计系统项目。

---

## 12. 最终判断

ArtDeco 的下一步重点不应是继续增加页面数量，而应优先提升：

- token 的表达能力
- 组件层级的清晰度
- 现有组件的稳定性
- 新建页面与新建组件的标准化能力

只有这样，ArtDeco 才能从“当前正在使用的前端样式体系”，进一步演化成 MyStocks 可持续扩展的前端平台。
