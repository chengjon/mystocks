# ArtDeco Standardization P2 Tasks

> **说明**
> 本文是 [`ARTDECO_STANDARDIZATION_PLAN.md`](ARTDECO_STANDARDIZATION_PLAN.md) 的 P2 可执行任务清单。
> P2 的定位不是继续做样板组件治理，而是把已经稳定下来的结构、样式和测试规则固化为 ArtDeco CLI 脚手架能力。

---

## 1. P2 目标

P2 要解决的问题是：

- 如何让新页面、新组件不再依赖人工复制旧文件
- 如何把已经稳定下来的规则固化为默认生成结果
- 如何让 ArtDeco 从“样板集合”升级为“可生成的平台”

P2 完成后，应具备：

- 自有 ArtDeco CLI 最小可用版本
- 稳定的组件 / 页面生成模板
- 与当前目录治理、token 规则、测试骨架一致的默认输出

---

## 2. P2 前置条件

只有当以下条件满足时，才适合正式启动 P2：

- P0 已完成：
  - token 缺口已识别
  - Base / Domain 边界已冻结
  - 样板组件名单已确定
- P1 已完成：
  - 首批样板组件已收敛
  - token 已补齐到足以支撑样板复制
  - 样板组件测试保护已建立

如果以上条件不满足，不应提前开发 CLI。

原因：

- 脚手架会过早固化不稳定结构
- 生成模板会反复返工
- 平台化会把历史混乱复制得更快

---

## 3. P2 工作原则

- CLI 只固化已经稳定的规则
- 生成器先解决高频场景，不追求一次覆盖所有页面
- 生成结果默认应满足当前治理规则
- CLI 不应引入新的平行结构
- 模板能力优先于花哨交互

---

## 4. P2 工作包

### 工作包 A：定义 CLI 范围与命令边界

目标：

- 明确 CLI 第一版做什么、不做什么

建议第一版支持：

- 生成 Base 组件
- 生成 Domain 组件
- 生成 ArtDeco 页面骨架
- 生成配套样式文件
- 生成最小测试骨架

第一版不建议支持：

- 全站重构
- 自动迁移旧组件
- 自动修复全部样式问题
- 跨域目录大规模搬迁

产物：

- CLI 范围定义
- 第一版命令集合说明

验收：

- 团队能明确 CLI 的职责边界
- CLI 不被误解成“自动重构工具”

### 工作包 B：固化组件生成模板

目标：

- 把 Base / Domain 组件的稳定结构变成标准模板

任务：

- 为 Base 组件确定统一模板
- 为 Domain 组件确定统一模板
- 明确样式文件、测试文件、导出方式和命名规则

产物：

- Base 组件模板
- Domain 组件模板

验收：

- 新生成组件的结构与现有样板一致
- 生成结果不需要大幅返工才能进入开发

### 工作包 C：固化页面骨架模板

目标：

- 把高频 ArtDeco 页面骨架标准化

任务：

- 统一页面目录结构
- 统一样式文件组织
- 统一 Request ID 展示位、加载态、错误态、空态的骨架
- 统一与 token 和基础组件的接入方式

产物：

- 页面骨架模板
- 页面生成参数清单

验收：

- 新页面生成后可直接进入业务开发
- 页面骨架天然符合现有治理风格

### 工作包 D：固化最小测试模板

目标：

- 让新组件 / 新页面从创建开始就具备最小回归保护入口

任务：

- 为组件模板补最小单测模板
- 为页面模板补最小交互或结构测试模板
- 定义哪些测试是默认生成的，哪些由业务自行补充

产物：

- 组件测试模板
- 页面测试模板

验收：

- 新生成产物默认具备基础测试入口
- 生成后不需要手工补齐最基本的测试骨架

### 工作包 E：实现 CLI 最小可用版本

目标：

- 提供可实际使用的第一版 ArtDeco CLI

任务：

- 选择 CLI 实现方式
- 实现模板渲染与文件输出
- 将生成路径对齐到现有目录治理规则
- 为 CLI 自身补最小验证方式

产物：

- CLI 第一版
- CLI 使用文档
- CLI 自测或样板验证方式

验收：

- 能稳定生成首批目标类型
- 生成路径、命名和文件内容符合仓库规范

---

## 5. 推荐命令粒度

第一版 CLI 建议保持简单，命令粒度不要过细。

建议命令类型：

- `generate base-component`
- `generate domain-component`
- `generate page`

可选扩展参数：

- 名称
- 目标域
- 是否带样式文件
- 是否带测试骨架

---

## 6. P2 不做什么

P2 阶段明确不做以下事情：

- 让 CLI 承担旧代码迁移
- 让 CLI 自动修复历史结构问题
- 在模板未稳定前支持大量高级参数
- 先做复杂交互，再补基础模板
- 用 CLI 规避原本应通过治理解决的问题

---

## 7. P2 完成标准

当以下条件同时成立时，可判定 P2 完成：

- CLI 范围边界清晰
- Base / Domain / Page 模板已固化
- 测试骨架已接入模板
- CLI 第一版可生成至少三类高频目标
- 生成结果符合目录、样式、测试与平台规范

---

## 8. 一句话总结

P2 的核心不是“把 CLI 做出来”，而是把已经稳定的 ArtDeco 平台规则可靠地固化进 CLI，避免继续靠手工复制传播结构。
