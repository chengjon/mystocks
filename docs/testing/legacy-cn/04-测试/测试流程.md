import re
from pydantic import field_validator, ValueError

@field_validator("password")
def validate_password(cls, v):
    # 密码长度至少8位
    if len(v) < 8:
        raise ValueError("Password must be at least 8 characters long")
    # 至少包含一个大写字母
    if not any(c.isupper() for c in v):
        raise ValueError("Password must contain at least one uppercase letter")
    # 至少包含一个特殊字符（非字母、非数字）
    if not re.search(r"[^a-zA-Z0-9]", v):
        raise ValueError("Password must contain at least one special character (!@#$%^&* etc.)")
    return v

from fastapi import HTTPException, status
from logging import getLogger

logger = getLogger(__name__)

# 自定义业务异常类
class BusinessException(HTTPException):
    def __init__(self, detail: str, status_code: int = status.HTTP_400_BAD_REQUEST):
        super().__init__(status_code=status_code, detail=detail)
        # 统一记录异常日志
        logger.error(f"业务异常：{detail}（状态码：{status_code}）")

# 全局通用异常处理（后续挂载到FastAPI app）
def register_exception_handlers(app):
    @app.exception_handler(BusinessException)
    async def business_exception_handler(request, exc):
        return JSONResponse(
            status_code=exc.status_code,
            content={"code": exc.status_code, "message": exc.detail, "data": None}
        )
二、第二步：向 AI 提出「补充审查要求」，解决「review 全部代码」的困惑
AI 现有报告的局限性是「未审查所有业务逻辑、未深入所有 API 端点、未完整查看测试文件」，而你要的「review 全部代码」并非 “无差别审查所有文件”（会导致信息过载、重点模糊），而是 **「全覆盖、分模块、有重点」的深度审查 **。
建议向 AI 下达以下具体、可落地的补充审查指令（直接复制使用），填补现有报告的空白，同时解决全量代码 review 的困惑：
plaintext
需求：基于现有MyStocks项目代码审查报告，补充完成「全量模块、重点聚焦」的深度代码审查，要求如下：
1.  审查范围：全覆盖（解决“未看全业务逻辑”的局限性）
   - 后端模块：按功能模块拆分审查（auth、market、indicators、system、database），每个模块至少覆盖核心路由、服务、仓库3类文件
   - 前端模块：按Vue组件拆分审查（页面组件、ArtDeco公共组件、api请求模块、路由、Pinia状态管理）
   - 测试模块：完整审查单元测试、集成测试、E2E测试文件，验证测试覆盖有效性与合理性
2.  审查重点：针对性填补现有报告空白（避免重复，聚焦未覆盖内容）
   - 后端：
     ①  深入所有API端点（尤其是404的/api/v1/market/*、/api/system/*），排查路由配置、请求方法、参数验证、响应格式的问题
     ②  数据库连接管理（PostgreSQL+TDengine）：排查连接池配置、异步/同步IO混用、连接泄露的具体问题，给出可落地的优化方案
     ③  业务逻辑层（services）：排查代码重复（尤其是CRUD操作）、依赖耦合的具体示例，给出抽离公共逻辑的方案
   - 前端：
     ①  Vue组件（尤其是页面空白相关的Home、Dashboard组件）：排查组件通信、生命周期、异步数据请求的潜在问题
     ②  ArtDeco 64个公共组件：排查组件复用性、props定义、样式隔离的问题
     ③  apiClient.ts与后端API的适配性：排查请求路径、参数格式、响应解析的一致性问题
   -  测试模块：
     ①  验证测试覆盖度：指出未覆盖的核心业务场景、API端点、组件功能
     ②  排查测试用例有效性：指出虚假通过、逻辑错误、无意义的测试用例
     ③  给出测试优化方案：如何补充缺失测试、如何统一测试风格
3.  审查方法论：要求「有示例、有细节、可落地」（解决“结论笼统”的问题）
   -  每个问题必须附带「具体文件路径+代码片段+问题分析」，避免模糊表述
   -  每个优化/修复建议必须附带「具体代码示例」，而非仅理论描述
   -  按「模块→文件→代码行」的层级拆解问题，便于定位和修复
4.  输出要求：结构化、分级化（便于优先级排序，避免信息过载）
   -  按「风险等级」（🔴 严重、🟡 中等、🟢 低风险）分级整理问题
   -  按「修复优先级」（紧急修复、近期优化、长期重构）分类给出行动建议
   -  每个模块单独生成「问题清单+修复方案」，最后汇总全项目的「整体优化路线图」
   -  补充「验证方法」：每个问题修复后，如何通过代码检查、测试用例验证修复效果
5.  额外要求：填补现有报告的局限性，给出明确结论
   -  明确回答：现有4个后端API（/api/v1/market/list等）404的具体原因（路由配置？控制器未注册？路径不匹配？）
   -  明确回答：Vue页面空白（挂载成功后）的前端业务逻辑/组件问题是否与后端API 404存在联动关系
   -  明确给出：全项目代码质量提升的「分步执行计划」（1周内、1个月内、3个月内）
三、第三步：后续行动流程（清晰有序，避免混乱）
第一阶段（1-2 天）：落地 AI 现有报告的「可靠结论」（紧急安全漏洞 + 快速优化项），修复后通过pm2 restart重启服务，用 E2E 测试脚本验证基础功能正常，无新增问题。
第二阶段（3-5 天）：等待 AI 返回补充审查报告，仔细研读「模块级问题清单 + 修复示例」，按「紧急修复→近期优化」的优先级，分模块推进修复（优先解决后端 API 404、Vue 组件逻辑问题）。
第三阶段（1-3 个月）：按照 AI 给出的「全项目优化路线图」，推进长期重构（如统一异常处理、抽离公共 CRUD 逻辑、优化数据库连接管理），每完成一个阶段，用单元测试 + E2E 测试验证效果，确保代码质量稳步提升。
关键提醒：
避免「一次性推进所有修复」，分模块、分优先级落地，防止引入新的 bug；
利用现有「测试基础设施」（单元测试、E2E 测试），每修复一个问题，都要运行对应测试用例，验证修复有效性；
对于 AI 补充报告中「存疑的结论」（如某段业务逻辑的优化建议），可结合项目实际业务场景判断是否采纳，无需完全照搬（AI 仅基于代码分析，可能不了解部分业务特殊逻辑）。
总结
AI 现有报告的「安全漏洞、配置问题」可靠且紧急，应优先采纳落地，快速降低系统风险；
「review 全部代码」的困惑，需通过「分模块、有重点、提具体要求」解决，避免全量无差别审查导致的信息过载；
后续行动遵循「先紧急、后优化、再重构」的原则，利用现有测试工具验证每一步修复，确保项目稳步推进。
按照这个流程，你既能快速解决当前的核心问题（安全风险、页面空白、API 404），又能逐步完成全项目的代码质量提升，彻底解决代码 review 的困惑。
