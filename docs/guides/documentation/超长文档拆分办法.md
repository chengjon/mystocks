## 1. 拆分原则与准备

### 1.1 核心原则

超长文件（1000+ 行）违反了「单一职责原则」，是维护噩梦。优化的核心是 **「按功能模块拆分，拆分为「父组件 + 多个子组件 + 辅助文件」」**。

| 原则 | 说明 | 示例 |
|------|------|------|
| **功能内聚** | 同一功能模块的模板、脚本、样式拆为一个子组件 | 交易统计卡片、订单列表 |
| **低耦合** | 子组件仅通过 props 接收数据，通过 emit 传递事件 | `props` + `defineEmits` |
| **粒度适中** | 不拆分过细（如单个按钮），也不保留过大子组件 | 控制在 500 行以内 |

### 1.2 拆分前准备

1. **统计文件行数**：确认是否需要拆分（建议 >800 行考虑拆分）
2. **识别功能模块**：用注释标记可拆分的逻辑区域
3. **规划目录结构**：确定子组件、API、composables 的存放位置
4. **备份原文件**：确保可回滚

```bash
# 统计 Vue 文件行数
wc -l src/views/artdeco-pages/*.vue

# 统计 Python 文件行数
wc -l web/backend/app/*.py
```

## 2. Vue 组件拆分

### 2.1 模板层面——拆分功能子组件

这是超长 Vue 文件的核心问题根源（模板语法错误难定位），优先拆分模板中的功能模块。

#### 2.1.1 识别功能模块

以 `ArtDecoTradingManagement.vue`（交易管理页面）为例：

| 功能模块 | 描述 | 拆分后子组件 |
|----------|------|--------------|
| 交易统计卡片区域 | 包含多个 `<ArtDecoStatCard>` | `ArtDecoTradingStats.vue` |
| 订单列表展示区域 | 包含订单查询、列表渲染、分页 | `ArtDecoTradingOrderList.vue` |
| 信号监控区域 | 实时信号、信号历史 | `ArtDecoSignalMonitoringOverview.vue` |
| 归因分析区域 | 收益归因、关键交易分析 | `ArtDecoAttributionAnalysis.vue` |

### 2.2 脚本层面——拆分复杂逻辑

#### 2.2.1 拆分 API 请求

将接口请求抽离到独立文件：

```typescript
// src/api/artdeco/trading.ts
import axios from '@/utils/axios'

export const getTradingStats = async (params: { date?: string }) => {
  return axios.get('/api/artdeco/trading/stats', { params })
}

export const getTradingSignals = async (params?: SignalFilter) => {
  return axios.get('/api/artdeco/trading/signals', { params })
}

export const getTradingHistory = async (params: PaginationParams) => {
  return axios.get('/api/artdeco/trading/history', { params })
}
```

#### 2.2.2 拆分工具方法

```typescript
// src/utils/artdeco/format.ts
export const formatProfit = (amount: number): string => {
  return `¥${amount.toFixed(2)}`
}

export const formatPercent = (value: number): string => {
  return `${value >= 0 ? '+' : ''}${value.toFixed(2)}%`
}
```

#### 2.2.3 拆分 Composables

按页面域组织 composables：

```
src/composables/artdeco/
├── useTradingData.ts       # 交易管理页面数据
├── useSignalMonitoring.ts  # 信号监控数据
└── useRiskAssessment.ts    # 风险评估数据
```

```typescript
// src/composables/artdeco/useTradingData.ts
import { ref, onMounted } from 'vue'
import { getTradingStats, getTradingSignals } from '@/api/artdeco/trading'
import { formatPercent } from '@/utils/artdeco/format'

export const useTradingData = () => {
  const tradingStats = ref([])
  const signals = ref([])
  const loading = ref(false)

  const loadTradingStats = async () => {
    loading.value = true
    try {
      const res = await getTradingStats()
      tradingStats.value = res.data
    } catch (error) {
      console.error('加载交易统计数据失败：', error)
    } finally {
      loading.value = false
    }
  }

  onMounted(() => {
    loadTradingStats()
  })

  return {
    tradingStats,
    signals,
    loading,
    loadTradingStats
  }
}
```

## 3. Python 后端拆分

> **适用场景**: 超长 Python 文件（1000+ 行），如 `risk_management.py`、`mystocks_complete.py`

### 3.1 拆分原则：领域驱动拆分

按业务领域划分，而非按代码行数机械拆分：

```
web/backend/app/api/v1/risk/
├── __init__.py           # 领域聚合器，统一导出
```

## 4. 设计规范合规性检查

> **重要性**: 拆分后组件必须符合 ArtDeco 设计系统规范，避免样式碎片化

### 4.1 必做清单

#### 4.1.1 导入设计令牌

```scss
@import '@/styles/artdeco-tokens.scss';
@import '@/styles/artdeco-patterns.scss';
```

#### 4.1.2 使用 CSS 变量替代硬编码颜色

```scss
/* ❌ 错误 */
color: #FF5252;
background: #141414;

/* ✅ 正确 */
color: var(--artdeco-up);           /* 涨 - 红色（中国A股） */
color: var(--artdeco-down);         /* 跌 - 绿色（中国A股） */
color: var(--artdeco-gold-primary); /* 金色强调 */
color: var(--artdeco-bg-card);      /* 卡片背景 */
color: var(--artdeco-fg-primary);   /* 主要文本 */
color: var(--artdeco-fg-muted);     /* 次要文本 */
```

#### 4.1.3 应用 ArtDeco 装饰混合

```scss
/* 角落装饰（L-shaped brackets） */
@include artdeco-geometric-corners(var(--artdeco-gold-primary), 16px, 2px);

/* 阶梯角（ziggurat effect） */
@include artdeco-stepped-corners(8px);

/* 悬停效果 */
@include artdeco-hover-lift-glow;
```

#### 4.1.4 遵循字体规范

```scss
/* 标题 - Marcellus 罗马字体 */
.component-title {
  font-family: var(--artdeco-font-heading);
  font-weight: var(--artdeco-font-bold);
}

/* 数字 - 等宽字体 */
.stat-value {
  font-family: var(--artdeco-font-mono);
  font-variant-numeric: tabular-nums;
}

/* 标签 - 大写 + 宽间距 */
.stat-label {
  text-transform: uppercase;
  letter-spacing: var(--artdeco-tracking-wide);
  font-size: var(--artdeco-text-xs);
}
```

### 4.2 检查清单

| 检查项 | 检查方法 | 状态 |
|--------|----------|------|
| 使用 `--artdeco-*` CSS 变量 | 搜索硬编码颜色值 | ☐ |
| 数字使用等宽字体 | 检查 `font-family` | ☐ |
| 标签使用大写 + 宽间距 | 检查 `text-transform` | ☐ |
| 容器应用角落装饰 | 检查 `@include artdeco-geometric-corners` | ☐ |
| 悬停效果一致 | 检查 `@include artdeco-hover-lift-glow` | ☐ |
| 导入 SCSS tokens | 检查 `@import` 语句 | ☐ |

## 6. 类型同步机制

> **适用场景**: 修改 Python API 后需要同步更新前端 TypeScript 类型

### 6.1 生成命令

```bash
# 基础生成（生成所有类型）
python scripts/generate_frontend_types.py

# 按域生成（推荐，仅生成指定模块）
python scripts/generate_frontend_types.py --domain risk

# 单文件生成
python scripts/generate_frontend_types.py --single --output web/frontend/src/api/artdeco/types.ts

# 生成 index.ts 导出
python scripts/generate_frontend_types.py --domain risk --generate-index
```

### 6.2 生成后检查

1. 检查类型文件是否存在
2. 验证 `index.ts` 导出是否正确
3. 运行 TypeScript 编译检查

```bash
npm run type-check
```

### 6.3 类型同步时机

| 操作 | 是否需要重新生成 |
|------|------------------|
| 修改 Python API 响应结构 | ✅ 是 |
| 新增 API 端点 | ✅ 是 |
| 修改字段类型 | ✅ 是 |
| 仅修改业务逻辑 | ❌ 否 |
| 修改前端组件样式 | ❌ 否 |

## 7. 拆分后验证流程

> **重要性**: 每次拆分完成后必须执行验证，确保代码质量

### 7.1 语法检查

```bash
# Python
python -m py_compile <拆分后的文件>

# Vue/TypeScript
npm run type-check
# 或
vue-tsc --noEmit
```

### 7.2 代码规范检查

```bash
# Python
ruff check <拆分后的文件>
mypy <拆分后的文件>

# Frontend
npm run lint
```

### 7.3 常见问题排查

| 问题代码 | 原因 | 解决方案 |
|----------|------|----------|
| `F821 Undefined name 'X'` | 导入路径错误或延迟导入 | 检查 `sys.path` 配置；添加 `# noqa: E402` |
| `E402 Module level import` | 导入在 `sys.path` 修改前 | 添加 `# noqa: E402` |
| `F401 imported but unused` | 保留导入但未使用 | 添加 `# noqa: F401` 或删除 |
| `F841 Local variable assigned but never used` | 变量未使用 | 删除或添加 `_` 前缀 |
| 组件样式异常 | 缺少 SCSS token 导入 | 添加 `@import` 语句 |
| TypeScript 类型错误 | API 类型未同步生成 | 运行 `npm run generate-types` |

### 7.4 运行测试

```bash
# Python 测试
pytest <相关测试文件>

# 前端测试
npm run test
```

### 7.5 验证清单

| 验证项 | 命令 | 状态 |
|--------|------|------|
| Python 语法正确 | `python -m py_compile *.py` | ☐ |
| Python linting 通过 | `ruff check .` | ☐ |
| TypeScript 编译通过 | `npm run type-check` | ☐ |
| 前端 linting 通过 | `npm run lint` | ☐ |
| 单元测试通过 | `pytest` / `npm run test` | ☐ |

## 8. 反模式警示

### 8.1 ❌ 禁止：拆分时保留注释代码

```python
# ❌ 错误 - 保留旧代码
def old_function():
    ...

# 旧代码已迁移至 core.py，但仍保留在此
# def another_old_function(): ...

# ✅ 正确 - 彻底删除
```

### 8.2 ❌ 禁止：硬编码颜色值

```scss
/* ❌ 错误 */
color: #FF5252;
background: #141414;
border-color: rgba(212, 175, 55, 0.5);

/* ✅ 正确 */
color: var(--artdeco-up);
background: var(--artdeco-bg-card);
border-color: var(--artdeco-border-default);
```

### 8.3 ❌ 禁止：忽略 linter 警告

```bash
# ❌ 错误 - 忽略 linting 结果
ruff check .  # 发现问题但不修复

# ✅ 正确 - 修复所有问题
ruff check . --fix
```

### 8.4 ❌ 禁止：组件样式不完整

```vue
<!-- ❌ 错误 - 缺少 scoped -->
<style>
.container { ... }
</style>

<!-- ✅ 正确 - 添加 scoped -->
<style scoped>
.container { ... }
</style>
```

### 8.5 ❌ 禁止：忘记导入设计令牌

```scss
/* ❌ 错误 - 无法使用 CSS 变量 */
<style scoped>
.card {
  color: var(--artdeco-fg-primary); /* 未定义 */
}
</style>

/* ✅ 正确 - 导入 tokens */
<style scoped lang="scss">
@import '@/styles/artdeco-tokens.scss';

.card {
  color: var(--artdeco-fg-primary);
}
</style>
```

### 8.6 ✅ 推荐：拆分后立即运行完整测试

```bash
# 确保拆分不破坏功能
pytest tests/
npm run test
```

## 9. 维护与审查

### 9.1 命名规范

| 类型 | 命名规则 | 示例 |
|------|----------|------|
| Vue 组件 | PascalCase + ArtDeco 前缀 | `ArtDecoTradingStats.vue` |
| API 文件 | 功能描述 + `.ts` | `trading.ts` |
| Composables | `use` + 功能描述 | `useTradingData.ts` |
| 工具函数 | 小写下划线 | `format_profit.ts` |

### 9.2 组件文档注释

每个子组件开头添加注释：

```vue
<!--
  ArtDecoTradingStats.vue
  
  功能：交易统计卡片组件，展示核心交易指标
  
  Props:
    - stats: 统计数据列表
  
  Events:
    - refresh: 刷新统计数据
  
  依赖:
    - ArtDecoStatCard 组件
-->
```

### 9.3 定期审查

结合 CI/CD 月度审查流程：

1. **行数检查**: 单个文件不超过 1000 行
2. **复杂度检查**: 组件逻辑复杂度
3. **复用率检查**: 评估组件复用价值
4. **规范检查**: 设计规范一致性

### 9.4 拆分好处总结

| 好处 | 说明 |
|------|------|
| **易定位错误** | 子组件体积小，语法错误、逻辑问题可快速定位 |
| **易维护扩展** | 新增功能只需创建新的子组件 |
| **可复用性高** | 拆分后的组件可在其他页面复用 |
| **便于测试** | 子组件可单独编写单元测试 |
| **样式一致** | 统一使用设计令牌，风格一致 |
