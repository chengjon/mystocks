## 1. 理论篇：为什么要整理

### 1.1 熵增定律与代码库

软件项目的目录结构天然趋向混乱 — 这是信息熵增的必然结果。每一次"临时放在根目录"、"先这样后面再整理"都是一次熵增事件。没有主动治理，混乱只会加速。

**混乱的成本是隐性的、累积的：**

| 影响维度 | 具体表现 | 隐性成本 |
|---------|---------|---------|
| 认知负荷 | 新人需要 30 分钟才能找到一个配置文件 | 每人每周浪费 2-5 小时 |
| 决策疲劳 | "这个文件放哪里？" 每次都要想 | 降低开发节奏，打断心流 |
| 重复劳动 | 同类文件散落各处，重复造轮子 | 代码膨胀，维护成本翻倍 |
| CI/CD 脆弱性 | 路径硬编码，移动文件就炸 | 部署失败，回滚成本 |
| 技术债利息 | 混乱滋生更多混乱 | 指数级恶化 |

### 1.2 整理的本质

目录整理不是"打扫卫生"，而是**信息架构设计** — 它回答的核心问题是：

> "当一个人（或机器）需要找到某个东西时，它应该在哪里？"

好的目录结构是一种**无声的文档**：它通过位置本身传达意图。Uncle Bob 称之为"Screaming Architecture" — 看一眼目录树，就知道这个项目是做什么的。

### 1.3 整理的时机

| 信号 | 严重程度 | 建议行动 |
|------|---------|---------|
| 根目录非隐藏文件 > 15 个 | ⚠️ 黄灯 | 安排整理 sprint |
| 根目录非隐藏文件 > 30 个 | 🔴 红灯 | 立即启动治理 |
| 新人找文件平均 > 5 分钟 | 🔴 红灯 | 结构需要重新设计 |
| 同类文件散落 3+ 个目录 | ⚠️ 黄灯 | 需要归并 |
| 目录层级 > 6 层 | ⚠️ 黄灯 | 过度嵌套，需要扁平化 |
| 存在 "misc"/"temp"/"old" 目录 | ⚠️ 黄灯 | 垃圾抽屉效应 |

### 1.4 整理的边界：什么不该动

> ⚠️ 整理有明确的"禁区"。并非项目中所有目录都在整理范围内。

现代项目中存在大量**工具链自动生成的隐藏目录**（以 `.` 开头），它们是各种开发工具、AI 编程助手、IDE 的运行时依赖。**移动、重命名或删除这些目录会导致工具链失效**。

详见 [第 4 章：禁区篇](#4-禁区篇工具链隐藏目录保护)。

## 2. 诊断篇：如何评估现状

整理之前，必须先做体检。盲目动手只会制造新的混乱。

### 2.1 定量诊断：数据采集

运行以下命令获取项目的"体检报告"：

```bash
# 1. 根目录文件数（健康值：≤15，仅统计非隐藏文件）
find . -maxdepth 1 -type f ! -name '.*' | wc -l

# 2. 顶层目录数（健康值：8-15，排除隐藏目录和缓存目录）
#    ⚠️ 隐藏目录（.开头）不计入统计 — 它们属于工具链，不在整理范围内
find . -maxdepth 1 -type d ! -name '.*' ! -name 'node_modules' ! -name '__pycache__' | wc -l

# 3. 文件类型分布（排除隐藏目录内的文件）
find . -type f -not -path '*/.*' -not -path './node_modules/*' | \
  grep -oE '\.[^.]+$' | sort | uniq -c | sort -rn | head -20

# 4. 各目录文件密度（找出膨胀点，排除隐藏目录）
find . -type f -not -path '*/.*' -not -path './node_modules/*' | \
  sed 's|/[^/]*$||' | sort | uniq -c | sort -rn | head -30

# 5. 最深嵌套层级（排除隐藏目录）
find . -type f -not -path '*/.*' | awk -F/ '{print NF-1}' | sort -rn | head -1

# 6. 孤儿文件检测（根目录中不属于标准类型的非隐藏文件）
find . -maxdepth 1 -type f ! -name '.*' ! -name 'README*' ! -name 'LICENSE*' \
  ! -name 'package.json' ! -name 'pyproject.toml' ! -name '*.lock' \
  ! -name 'Makefile' ! -name 'Dockerfile' ! -name '*.toml' ! -name '*.cfg'

# 7. 隐藏目录清单（仅供了解，不纳入整理范围）
echo "=== 工具链隐藏目录（禁止整理）==="
find . -maxdepth 1 -type d -name '.*' ! -name '.git' | sort
```

### 2.2 定性诊断：结构审查清单

对照以下清单逐项检查：

- [ ] **命名一致性**：目录名是否遵循统一的命名规范（kebab-case / snake_case / camelCase）？
- [ ] **职责单一性**：每个目录是否只承担一种职责？
- [ ] **层级合理性**：是否存在只有一个子目录的目录（无意义嵌套）？
- [ ] **对称性**：同级目录的粒度是否一致（不会一个目录 500 文件，旁边一个目录 2 文件）？
- [ ] **可发现性**：新人能否在 3 次点击内找到任意文件？
- [ ] **README 覆盖率**：关键目录是否有 README 说明其用途？
- [ ] **.gitignore 完整性**：生成物（var/log/reports/cache）是否被正确忽略？
- [ ] **死文件检测**：是否存在 6 个月以上未修改且无引用的文件？
- [ ] **隐藏目录完整性**：工具链隐藏目录是否完好无损、未被误移动？

### 2.3 诊断报告模板

```markdown
## 项目目录健康度报告
- 日期：YYYY-MM-DD
- 根目录文件数：XX（健康值 ≤15，不含隐藏文件）→ 🔴/🟡/🟢
- 顶层目录数：XX（健康值 8-15，不含隐藏目录）→ 🔴/🟡/🟢
- 隐藏目录数：XX（仅记录，不纳入评分）
- 最深嵌套层级：X（健康值 ≤5）→ 🔴/🟡/🟢
- 文件总数：XXXX
- 最大目录文件密度：XXX 文件/目录
- 命名一致性：XX%
- 死文件比例：XX%
- 综合评分：X/10
```

## 3. 原则篇：整理的核心法则

### 3.1 七大核心原则

#### 原则一：根目录极简主义 (Root Minimalism)

> 根目录是项目的"门面"，只放"入口"和"顶级配置"。

**允许存在于根目录的文件（白名单）：**
- 项目入口：`README.md`, `LICENSE`, `CHANGELOG.md`
- 包管理：`package.json`, `pyproject.toml`, `requirements.txt`, `go.mod`
- 工具链配置：`.eslintrc`, `tsconfig.json`, `Makefile`, `Dockerfile`
- VCS 配置：`.gitignore`, `.gitattributes`
- 隐藏文件/目录：所有 `.` 开头的文件和目录（工具链自管理，见原则七）

**绝不应出现在根目录的文件：**
- 任何 `TASK-*.md`、`TODO-*.md` 等工作文档
- 任何 `*.log`、`*.csv`、`*.json` 数据文件
- 任何 `test_*.py`、`check_*.sh` 等脚本
- 任何 `backup_*`、`old_*`、`temp_*` 临时文件

#### 原则二：按功能聚类 (Function-Based Clustering)

> 文件按"做什么"分组，而非按"是什么"分组。

```
❌ 按类型分组（反模式）：
├── python_files/
├── markdown_files/
├── yaml_files/

✅ 按功能分组：
├── scripts/dev/        # 开发辅助
├── scripts/deploy/     # 部署相关
├── docs/guides/        # 使用指南
├── docs/architecture/  # 架构设计
├── config/             # 所有配置
```

#### 原则三：路径无关性 (Path Independence)

> 任何脚本/代码都不应依赖"从哪里执行"，而应基于项目根目录动态解析路径。

```python
# ✅ 正确：基于文件位置动态计算
from pathlib import Path
PROJECT_ROOT = Path(__file__).resolve().parent.parent.parent

# ❌ 错误：依赖执行位置
with open("config/settings.yaml") as f: ...

# ✅ 正确：基于项目根的绝对路径
with open(PROJECT_ROOT / "config" / "settings.yaml") as f: ...
```

#### 原则四：自治边界 (Autonomous Boundaries)

> 子系统（如 web/frontend、services/api）拥有自己的目录自治权，不受全局规则约束。

这意味着：
- `web/frontend/` 可以有自己的 `docs/`、`tests/`、`config/`
- 全局的 `check-structure.sh` 应排除子系统路径
- 子系统内部遵循自己的（或框架的）约定

#### 原则五：可预测性 (Predictability)

> 给定一个文件的描述，任何团队成员都应该能猜到它在哪里。

测试方法：随机选 5 个文件描述，让团队成员猜路径。如果正确率 < 80%，结构需要优化。

#### 原则六：渐进式演化 (Progressive Evolution)

> 目录结构应该随项目成长而演化，但每次变更都应该是可控的、可回滚的。

不要追求"一步到位的完美结构"。好的结构是迭代出来的，不是设计出来的。

#### 原则七：工具链免疫 (Toolchain Immunity) ⚠️ v2.1 新增

> 所有以 `.` 开头的隐藏目录和文件，默认属于工具链管辖，整理时必须跳过。

这是现代开发的"零号原则"：
- 隐藏目录是各种工具（IDE、AI 助手、VCS、构建系统）的运行时状态存储
- 移动或删除它们会导致工具链配置丢失、会话中断、缓存失效
- 它们的内部结构由工具自身管理，人工干预只会制造问题
- 即使某些隐藏目录看起来"很大"或"很乱"，也不应纳入整理范围

详见 [第 4 章：禁区篇](#4-禁区篇工具链隐藏目录保护)。

### 3.2 原则优先级

当原则之间冲突时，按以下优先级裁决：

```
工具链免疫 > 路径无关性 > 根目录极简 > 按功能聚类 > 可预测性 > 自治边界 > 渐进演化
```

理由：工具链免疫是绝对红线（破坏它会导致开发环境崩溃），路径无关性是技术基础（破坏它会导致运行时错误），根目录极简是最高可见度的改善。

## 4. 禁区篇：工具链隐藏目录保护

> ⚠️ 本章是 v2.1 的核心新增内容。现代项目中，工具链隐藏目录的数量和重要性已远超传统认知。

### 4.1 为什么需要专门一章

2024-2026 年间，AI 编程工具爆发式增长。一个典型的现代项目可能同时使用 10+ 种工具，每种工具都会在项目根目录下创建自己的 `.xxx/` 隐藏目录。这些目录：

- **数量多**：一个活跃项目可能有 20+ 个隐藏目录
- **功能关键**：存储会话状态、配置、缓存、索引等运行时数据
- **互不兼容**：每个工具有自己的目录结构约定，不可混淆
- **静默依赖**：工具启动时自动检测这些目录，缺失则功能降级或失效

### 4.2 隐藏目录分类图谱

| 类别 | 典型目录 | 用途 | 移动后果 |
|------|---------|------|---------|
| **AI 编程助手** | `.claude/`, `.amazonq/`, `.cursor/`, `.gemini/`, `.opencode/`, `.zencoder/`, `.zenflow/`, `.omc/`, `.specify/` | AI 会话记忆、项目上下文、提示词配置、MCP 设置 | 丢失所有 AI 对项目的理解和历史对话，需重新建立上下文 |
| **IDE / 编辑器** | `.vscode/`, `.idea/`, `.fleet/` | 工作区设置、调试配置、扩展配置、任务定义 | 丢失团队共享的编辑器配置、调试启动项、推荐扩展列表 |
| **版本控制** | `.git/`, `.github/`, `.gitlab/` | 仓库数据、CI/CD 工作流、Issue 模板 | `.git/` 移动 = 仓库损毁；`.github/` 移动 = CI/CD 失效 |
| **构建 / 缓存** | `.mypy_cache/`, `.pytest_cache/`, `.ruff_cache/`, `__pycache__/` | 类型检查缓存、测试缓存、Lint 缓存、字节码缓存 | 下次运行时重建（性能损失），但不会永久损坏 |
| **任务管理** | `.taskmaster/`, `.todo/` | AI 任务分解、进度追踪 | 丢失任务上下文和进度状态 |
| **测试工具** | `.playwright-mcp/`, `.benchmarks/` | 浏览器自动化配置、性能基准数据 | 测试环境配置丢失，基准数据不可恢复 |
| **项目配置** | `.config/`, `.shared/`, `.env`, `.env.*` | 项目级配置覆盖、共享资源、环境变量 | 运行时配置丢失，服务无法启动 |
| **迁移 / 归档** | `.migration/`, `.archive/` | 迁移状态记录、归档索引 | 迁移进度丢失，归档索引断裂 |
| **Git 扩展** | `.worktrees/`, `.gitattributes`, `.gitignore` | 多工作树、文件属性、忽略规则 | 多分支并行开发能力丢失 |

### 4.3 核心规则：三不原则

```
🚫 不移动 — 隐藏目录必须留在工具期望的位置（通常是项目根目录）
🚫 不重命名 — 目录名是工具硬编码的查找路径，改名等于删除
🚫 不合并 — 不同工具的隐藏目录绝不能合并（如把 .claude/ 合并到 .ai/）
```

**唯一允许的操作：**
- ✅ 在 `.gitignore` 中忽略不需要版本控制的隐藏目录
- ✅ 删除已确认不再使用的工具的隐藏目录（如卸载了 Cursor 后删除 `.cursor/`）
- ✅ 清理缓存类隐藏目录的内容（如 `.mypy_cache/`），但不删除目录本身
- ✅ 编辑隐藏目录内的配置文件（如 `.vscode/settings.json`），但不改变目录结构

### 4.4 .gitignore 策略

隐藏目录分为"应该版本控制"和"不应该版本控制"两类：

```gitignore
# === 应该版本控制的隐藏目录 ===
# （团队共享配置，提交到仓库）
# .github/          — CI/CD 工作流、Issue 模板
# .vscode/          — 共享的编辑器设置（settings.json, extensions.json, launch.json）
# .claude/          — 项目级 AI 配置（如 CLAUDE.md 引用的设置）

# === 不应该版本控制的隐藏目录 ===
# （个人状态/缓存，加入 .gitignore）
.mypy_cache/
.pytest_cache/
.ruff_cache/
__pycache__/
.coverage
.env
.env.*
!.env.example

# AI 工具的会话/缓存数据（因人而异）
.claude-trace/
.amazonq/
.cursor/
.gemini/
.opencode/
.omc/
.zencoder/
.zenflow/
.specify/

# 构建产物
.worktrees/
.benchmarks/
```

**判断标准**：
- 如果目录内容是**团队共享的配置** → 版本控制
- 如果目录内容是**个人状态或缓存** → `.gitignore`
- 如果不确定 → 先 `.gitignore`，需要时再加入版本控制

### 4.5 诊断时如何处理隐藏目录

在进行项目目录健康度诊断时，隐藏目录应该：

| 诊断环节 | 处理方式 |
|---------|---------|
| 统计根目录文件/目录数 | **排除**隐藏目录，不计入健康度指标 |
| 文件类型分布分析 | **排除**隐藏目录内的文件 |
| 目录深度分析 | **排除**隐藏目录的嵌套层级 |
| 磁盘空间分析 | **单独列出**隐藏目录的空间占用，但不计入"需要整理"的范围 |
| 死文件检测 | **排除**隐藏目录内的文件 |
| 命名规范检查 | **排除**隐藏目录（它们的命名由工具决定） |

```bash
# 正确的诊断命令模式：始终排除隐藏目录
find . -maxdepth 1 -type d ! -name '.*' ! -name 'node_modules' | wc -l  # ✅
find . -maxdepth 1 -type d | wc -l                                       # ❌ 会把隐藏目录也算进去

# 如果需要了解隐藏目录的情况，单独统计
echo "=== 隐藏目录空间占用 ==="
du -sh .*/  2>/dev/null | sort -rh | head -20
```

### 4.6 新工具引入时的处理流程

当团队引入新的开发工具时：

```
1. 确认工具会创建哪些隐藏目录/文件
   → 查阅工具文档或试运行后观察

2. 判断是否需要版本控制
   → 团队共享配置 → 提交
   → 个人状态/缓存 → .gitignore

3. 更新 .gitignore
   → 添加对应条目，附注释说明是哪个工具

4. 通知团队
   → 在 CONTRIBUTING.md 或 README 中记录

5. 验证
   → 确认工具在 clone 后能正常初始化
   → 确认 .gitignore 规则不会误排除需要的文件
```

### 4.7 常见误操作与恢复

| 误操作 | 症状 | 恢复方法 |
|--------|------|---------|
| 删除了 `.vscode/` | 团队编辑器配置丢失 | `git checkout -- .vscode/` |
| 移动了 `.claude/` | AI 助手丢失项目上下文 | 移回原位，或让工具重新初始化 |
| 把 `.github/` 重命名为 `github/` | CI/CD 全部失效 | `git mv github/ .github/` |
| 清理脚本误删 `.env` | 服务无法启动 | 从 `.env.example` 重建，或从备份恢复 |
| 合并了多个 AI 工具目录 | 所有 AI 工具配置损坏 | 删除合并目录，让各工具重新初始化 |
| 把 `.git/` 加入了整理范围 | 仓库损毁 | 从远程重新 clone |

## 5. 分类篇：文件归属的决策模型

### 5.1 三级决策树

面对任何一个文件，按以下顺序判断其归属：

```
Level 0: 是否为隐藏文件/目录（.开头）？
  ├─ 是 → 🚫 不整理，属于工具链管辖（见第4章）
  └─ 否 → 进入 Level 1

Level 1: 它属于哪个子系统？
  ├─ 属于 web/frontend → 由前端自治管理
  ├─ 属于 web/backend  → 由后端自治管理
  ├─ 属于 services/*   → 由对应服务自治管理
  └─ 属于全局 → 进入 Level 2

Level 2: 它的功能角色是什么？
  ├─ 源代码（业务逻辑）    → src/
  ├─ 配置文件              → config/
  ├─ 脚本（开发/部署/维护） → scripts/{dev,deploy,maintenance}/
  ├─ 文档                  → docs/{guides,architecture,api,standards}/
  ├─ 测试                  → tests/{unit,integration,e2e}/
  ├─ 生成物（报告/日志）    → reports/ 或 var/log/（应 .gitignore）
  └─ 无法分类              → 进入 Level 3

Level 3: 特殊处理
  ├─ 临时文件 → temp/（定期清理）
  ├─ 历史遗留 → archived/（标注日期）
  └─ 真的无法分类 → 提出讨论，更新分类规则
```

### 5.2 文件类型速查矩阵

| 文件类型 | 目标位置 | 示例 |
|---------|---------|------|
| `.py` 业务代码 | `src/模块名/` | `src/strategy/backtest.py` |
| `.py` 脚本工具 | `scripts/dev/` 或 `scripts/deploy/` | `scripts/dev/check_imports.py` |
| `.py` 测试文件 | `tests/unit/` 或 `tests/integration/` | `tests/unit/test_backtest.py` |
| `.md` 用户指南 | `docs/guides/` | `docs/guides/getting-started.md` |
| `.md` 架构文档 | `docs/architecture/` | `docs/architecture/data-flow.md` |
| `.md` API 文档 | `docs/api/` | `docs/api/rest-endpoints.md` |
| `.md` 标准规范 | `docs/standards/` | `docs/standards/naming-convention.md` |
| `.md` 任务/报告 | `docs/reports/` | `docs/reports/2025-01-sprint-review.md` |
| `.yaml/.yml` | `config/` | `config/docker-compose.yml` |
| `.json` 配置 | `config/` 或项目根（如 `package.json`） | `config/eslint.json` |
| `.json` 数据 | `data/` 或 `reports/` | `data/fixtures/sample.json` |
| `.sh` 脚本 | `scripts/{dev,deploy,maintenance}/` | `scripts/deploy/release.sh` |
| `.sql` | `database/migrations/` 或 `scripts/database/` | `database/migrations/001_init.sql` |
| `.env*` | 项目根目录（隐藏文件，不移动） | `.env`, `.env.example` |
| `Dockerfile*` | 项目根或 `docker/` | `docker/Dockerfile.prod` |
| `*.log` | `var/log/`（应 .gitignore） | `var/log/app.log` |
| `*.bak`, `*.old` | 删除或 `archived/` | 优先删除，VCS 已有历史 |
| 隐藏目录 `.xxx/` | **原地不动** ⚠️ | `.claude/`, `.vscode/` |
| 隐藏文件 `.xxx` | **原地不动** ⚠️ | `.env`, `.gitignore`, `.eslintrc` |

### 5.3 边界情况处理

**Q: 一个文件同时属于两个分类怎么办？**
A: 按"主要用途"分类。例如 `deploy-config.yaml` 虽然是配置文件，但主要服务于部署，放 `config/deploy/` 或 `scripts/deploy/`。

**Q: 子系统内的文件要不要遵循全局规则？**
A: 不强制。子系统遵循自己的框架约定（如 Next.js 的 `app/`、`components/`）。

**Q: README 文件放哪里？**
A: README 永远不移动。每个目录的 README.md 就地保留，它是该目录的"门牌"。

**Q: 隐藏文件（如 `.env`）算根目录文件吗？**
A: 统计根目录健康度时不计入。隐藏文件属于工具链范畴，不纳入整理评估。

**Q: `node_modules/`、`__pycache__/` 等缓存目录要整理吗？**
A: 不整理。它们是包管理器/运行时的产物，应在 `.gitignore` 中忽略。如果它们出现在非预期位置（如 `bak/old_project/node_modules/`），说明备份目录本身需要清理。

## 6. 架构篇：目录结构设计范式

### 6.1 四种主流范式

#### 范式 A：按层分组 (Layer-Based)

```
src/
├── controllers/    # 控制层
├── services/       # 业务层
├── models/         # 数据层
├── utils/          # 工具层
└── middleware/     # 中间件
```

**适用场景**：小型项目（< 50 文件），团队对 MVC 模式熟悉
**优点**：简单直观，上手快
**缺点**：功能分散，修改一个功能要跨多个目录

#### 范式 B：按功能分组 (Feature-Based)

```
src/
├── auth/           # 认证模块
│   ├── controller.py
│   ├── service.py
│   ├── model.py
│   └── tests/
├── trading/        # 交易模块
│   ├── controller.py
│   ├── service.py
│   ├── model.py
│   └── tests/
└── shared/         # 共享模块
```

**适用场景**：中大型项目，微服务架构
**优点**：高内聚低耦合，功能自包含
**缺点**：共享代码的归属容易模糊

#### 范式 C：混合分组 (Hybrid)

```
src/
├── core/           # 核心业务（按功能）
│   ├── auth/
│   └── trading/
├── infrastructure/ # 基础设施（按层）
│   ├── database/
│   ├── cache/
│   └── messaging/
└── shared/         # 共享工具
    ├── utils/
    └── types/
```

**适用场景**：大型项目，需要平衡灵活性和规范性
**优点**：兼顾两种范式的优点
**缺点**：需要明确的边界定义

#### 范式 D：Feature-Sliced Design (FSD)

```
src/
├── app/            # 应用初始化、全局配置
├── processes/      # 跨页面业务流程
├── pages/          # 页面组件
├── widgets/        # 独立 UI 块
├── features/       # 用户交互功能
├── entities/       # 业务实体
└── shared/         # 共享基础设施
```

**适用场景**：大型前端项目
**优点**：严格的依赖方向（上层可依赖下层，反之不行）
**缺点**：学习曲线陡峭

### 6.2 范式选择决策树

```
项目文件数 < 50？
  ├─ 是 → 范式 A（按层）
  └─ 否 → 是否有明确的功能模块边界？
           ├─ 是 → 范式 B（按功能）
           └─ 否 → 是否是前端项目？
                    ├─ 是 → 范式 D（FSD）
                    └─ 否 → 范式 C（混合）
```

### 6.3 通用骨架模板

无论选择哪种范式，以下顶层结构是通用的：

```
project-root/
│
│── .claude/              # 🔒 AI 工具（禁止整理）
│── .vscode/              # 🔒 IDE 配置（禁止整理）
│── .github/              # 🔒 CI/CD 工作流（禁止整理）
│── .其他工具目录/          # 🔒 各种工具链目录（禁止整理）
│
├── src/                  # 源代码（选择上述范式之一组织）
├── tests/                # 测试代码
│   ├── unit/
│   ├── integration/
│   └── e2e/
├── config/               # 配置文件集中管理
├── scripts/              # 脚本工具
│   ├── dev/              # 开发辅助
│   ├── deploy/           # 部署相关
│   └── maintenance/      # 维护脚本
├── docs/                 # 文档
│   ├── guides/           # 使用指南
│   ├── architecture/     # 架构设计
│   ├── standards/        # 规范标准
│   └── api/              # API 文档
├── data/                 # 静态数据/fixtures
├── archived/             # 历史归档（标注日期）
│
├── README.md             # 项目说明
```

## 7. 执行篇：安全迁移方法论

### 7.1 迁移策略选择

| 策略 | 适用场景 | 风险 | 耗时 |
|------|---------|------|------|
| **Big Bang** | 小项目，团队 ≤ 3 人 | 高（一次性大量变更） | 短 |
| **渐进式** | 中大型项目，持续交付 | 低（每次小步变更） | 长 |
| **分支隔离** | 有 CI/CD，需要验证 | 中（合并冲突风险） | 中 |

**推荐**：绝大多数情况下选择**渐进式**。

### 7.2 渐进式迁移七步法

#### Step 1: 快照 — 记录现状

```bash
# 生成当前结构快照（排除隐藏目录和缓存）
find . -not -path '*/.*' -not -path './node_modules/*' \
  -not -path './__pycache__/*' | sort > /tmp/structure_before.txt

# 记录隐藏目录清单（仅供参考，不纳入迁移范围）
echo "=== 隐藏目录（不迁移）===" > /tmp/dotdirs_inventory.txt
find . -maxdepth 1 -type d -name '.*' >> /tmp/dotdirs_inventory.txt

# 确保 VCS 干净
git status  # 应该是 clean
git tag pre-restructure-$(date +%Y%m%d)  # 打标签用于回滚
```

#### Step 2: 骨架 — 创建目标目录

```bash
# 创建目标目录结构（不移动任何文件）
mkdir -p src config scripts/{dev,deploy,maintenance} \
  docs/{guides,architecture,standards,api,reports} \
  tests/{unit,integration,e2e} data archived

# 提交空目录结构（用 .gitkeep）
find src config scripts docs tests data archived -type d -empty \
  -exec touch {}/.gitkeep \;
git add -A && git commit -m "chore: create target directory skeleton"
```

#### Step 3: 分类 — 制定迁移清单

```bash
# 列出所有需要迁移的文件（排除隐藏文件/目录）
find . -maxdepth 1 -type f ! -name '.*' ! -name 'README*' \
  ! -name 'LICENSE*' ! -name 'package.json' ! -name 'pyproject.toml' \
  ! -name '*.lock' ! -name 'Makefile' ! -name 'Dockerfile' \
  ! -name '*.toml' ! -name '*.cfg' ! -name 'CLAUDE.md' \
  | sort > /tmp/files_to_migrate.txt

# 按类型分组，制定迁移映射表
# 格式：源路径 → 目标路径
```

> ⚠️ 迁移清单中绝不应出现任何 `.` 开头的文件或目录。如果出现，说明过滤条件有误。

#### Step 4: 试移 — 小批量验证

```bash
# 先移动 3-5 个低风险文件
git mv some_script.py scripts/dev/some_script.py

# 立即验证
python -c "import scripts.dev.some_script"  # 或对应的验证方式
git commit -m "chore: migrate some_script.py to scripts/dev/"
```

#### Step 5: 批移 — 按类型批量迁移

```bash
# 按类型分批，每批提交一次
# 第 1 批：文档类（最低风险）
git mv *.md docs/reports/ 2>/dev/null  # 注意排除根目录的 README.md 等

# 第 2 批：脚本类
git mv check_*.sh scripts/dev/
git mv deploy_*.sh scripts/deploy/

# 第 3 批：配置类
git mv *.yaml config/
git mv *.yml config/

# 第 4 批：源代码（最高风险，需要更新 import）
# ...逐个处理，每个都验证
```

#### Step 6: 修复 — 更新引用路径

```bash
# 搜索所有硬编码路径（排除隐藏目录）
grep -rn "old_path" --include="*.py" --include="*.sh" --include="*.yaml" \
  --exclude-dir=".*" --exclude-dir="node_modules" .

# 批量替换（谨慎操作）
find . -name "*.py" -not -path "*/.*" -exec \
  sed -i 's|from old_module|from src.new_module|g' {} \;

# 更新 CI/CD 配置中的路径
# 更新 Makefile 中的路径
# 更新 Docker 构建上下文中的路径
```

#### Step 7: 验证 — 全面检查

```bash
# 1. 运行测试套件
pytest tests/ -v

# 2. 运行构建
make build  # 或对应的构建命令

# 3. 对比结构快照（排除隐藏目录）
find . -not -path '*/.*' -not -path './node_modules/*' \
  | sort > /tmp/structure_after.txt
diff /tmp/structure_before.txt /tmp/structure_after.txt

# 4. 验证隐藏目录未被触动
diff /tmp/dotdirs_inventory.txt <(find . -maxdepth 1 -type d -name '.*' | sort)
# 应该无差异

# 5. 检查根目录文件数
echo "根目录非隐藏文件数："
find . -maxdepth 1 -type f ! -name '.*' | wc -l
```

### 7.3 回滚预案

```bash

git reset --hard pre-restructure-20250101
git clean -fd  # 清理未跟踪文件
```

## 8. 防御篇：建立长效防线

> 整理一次不难，难的是不再乱回去。

### 8.1 三道防线模型

```
第一道：规范（人）  →  团队共识 + 文档
第二道：自动化（机器）  →  Git Hook + CI
第三道：定期审计（制度）  →  周检 + 月报
```

### 8.2 第一道防线：规范文档

- 维护一份 `FILE_ORGANIZATION_RULES.md` 作为 Single Source of Truth
- 新成员 onboarding 必读
- 每次整理后更新规范

### 8.3 第二道防线：Git Hook + CI

**pre-commit hook（含 dot-directory 保护）：**

```bash
#!/bin/bash
# .githooks/pre-commit

MAX_ROOT_FILES=20
ERRORS=0

# ── 1. 根目录文件数检查（排除 dot-files）──
root_count=$(git diff --cached --name-only | grep -c '^[^./][^/]*$')
if [ "$root_count" -gt "$MAX_ROOT_FILES" ]; then
  echo "⚠️  根目录文件数 ($root_count) 超过阈值 ($MAX_ROOT_FILES)"
  ERRORS=$((ERRORS + 1))
fi

# ── 2. 禁止移动/删除 dot-directories ──
moved_dots=$(git diff --cached --name-status | grep -E '^[RD].*\s+\.' | grep -v '.gitignore')
if [ -n "$moved_dots" ]; then
  echo "🚫 检测到 dot-directory/dot-file 被移动或删除："
  echo "$moved_dots"
  echo "   dot-目录是工具链配置，不应被整理。"
  ERRORS=$((ERRORS + 1))
fi

# ── 3. 文件类型放置检查 ──
wrong_py=$(git diff --cached --name-only | grep -E '^[^/]+\.py$' | grep -v -E '^(conftest|setup|manage)\.py$')
if [ -n "$wrong_py" ]; then
  echo "⚠️  Python 文件不应直接放在根目录："
  echo "$wrong_py"
  ERRORS=$((ERRORS + 1))
fi

wrong_sh=$(git diff --cached --name-only | grep -E '^[^/]+\.sh$')
if [ -n "$wrong_sh" ]; then
  echo "⚠️  Shell 脚本不应直接放在根目录："
  echo "$wrong_sh"
  ERRORS=$((ERRORS + 1))
fi

# ── 4. 禁止提交临时文件 ──
temp_files=$(git diff --cached --name-only | grep -iE '\.(log|tmp|bak|swp|orig)$')
if [ -n "$temp_files" ]; then
  echo "🚫 临时文件不应提交："
  echo "$temp_files"
  ERRORS=$((ERRORS + 1))
fi

if [ "$ERRORS" -gt 0 ]; then
  echo ""
  echo "提交被阻止。请修正以上问题后重试。"
  exit 1
fi
```

**CI Pipeline 检查：**

```yaml
# .github/workflows/structure-check.yml
name: Structure Check
on: [pull_request]
jobs:
  check-structure:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Check root cleanliness
        run: |
          # 统计根目录非 dot 文件数
          count=$(find . -maxdepth 1 -type f ! -name '.*' | wc -l)
          echo "Root files: $count"
          if [ "$count" -gt 25 ]; then
            echo "::error::根目录文件数 ($count) 超标"
            exit 1
          fi
      - name: Verify dot-directories intact
        run: |
          # 检查关键 dot-dirs 是否存在
          for dir in .git .github .vscode; do
            if [ ! -d "$dir" ]; then
              echo "::error::关键目录 $dir 缺失"
              exit 1
            fi
          done
      - name: Check file placement
        run: |
          # 检查是否有脚本/报告误放根目录
          violations=$(find . -maxdepth 1 -type f \
            \( -name '*.py' -o -name '*.sh' -o -name '*.log' \) \
            ! -name 'conftest.py' ! -name 'setup.py' ! -name 'manage.py' \
            2>/dev/null)
          if [ -n "$violations" ]; then
            echo "::error::以下文件不应在根目录: $violations"
            exit 1
          fi
```

### 8.4 第三道防线：定期审计

| 频率 | 动作 | 工具 |
|------|------|------|
| 每日 | 自动清理 `*.log`, `*.tmp` | cron + find -delete |
| 每周 | 运行 check-structure.sh | CI scheduled job |
| 每月 | 生成 DHI 报告，对比趋势 | 自定义脚本 |
| 每季 | 全面审查，更新规范文档 | 人工 review |

**审计脚本模板：**

```bash
#!/bin/bash
# scripts/dev/audit-structure.sh

echo "=== 目录健康度审计 $(date +%Y-%m-%d) ==="

# 根目录文件数（排除 dot-files）
root_files=$(find . -maxdepth 1 -type f ! -name '.*' | wc -l)
echo "根目录文件数: $root_files (阈值: ≤20)"

# 根目录目录数（排除 dot-dirs）
root_dirs=$(find . -maxdepth 1 -type d ! -name '.*' ! -name '.' | wc -l)
echo "根目录目录数: $root_dirs (阈值: ≤15)"

# dot-directories 数量（仅统计，不整理）
dot_dirs=$(find . -maxdepth 1 -type d -name '.*' ! -name '.' ! -name '.git' | wc -l)
echo "Dot-directories: $dot_dirs (仅监控，不整理)"

# 最大嵌套深度
max_depth=$(find . -type f -not -path './.git/*' -not -path '*/node_modules/*' \
  | awk -F/ '{print NF-1}' | sort -rn | head -1)
echo "最大嵌套深度: $max_depth (阈值: ≤6)"

# 空目录
empty_dirs=$(find . -type d -empty -not -path './.git/*' | wc -l)
echo "空目录数: $empty_dirs"

# 大文件
large_files=$(find . -type f -size +10M -not -path './.git/*' -not -path '*/node_modules/*' | wc -l)
echo "大文件(>10MB): $large_files"

echo "=== 审计完成 ==="
```

## 9. 度量篇：DHI 目录健康度指标

### 9.1 六维度评分体系

| 维度 | 权重 | 满分条件 | 扣分规则 |
|------|------|----------|----------|
| 根目录整洁度 | 25% | 非 dot 文件 ≤15，非 dot 目录 ≤12 | 每超 5 个扣 10 分 |
| 命名一致性 | 15% | 100% 符合命名规范 | 每个违规扣 5 分 |
| 嵌套合理性 | 15% | 最大深度 ≤6 | 每超 1 层扣 10 分 |
| 分类准确性 | 20% | 100% 文件在正确位置 | 每个错位扣 3 分 |
| 冗余度 | 15% | 无重复/废弃文件 | 每个冗余扣 5 分 |
| 工具链完整性 | 10% | dot-dirs 完整且功能正常 | 每个缺失/损坏扣 15 分 |

> 注意：dot-directories 不计入"根目录整洁度"的统计，但"工具链完整性"维度专门评估它们的健康状态。

### 9.2 评分公式

```
DHI = Σ(维度得分 × 权重)

等级：
  A (90-100): 优秀 — 结构清晰，维护成本低
  B (75-89):  良好 — 小问题，定期维护即可
  C (60-74):  及格 — 需要专项整理
  D (40-59):  较差 — 需要系统性重组
  F (<40):    危险 — 结构严重失控
```

### 9.3 趋势追踪

```markdown
| 日期 | DHI | 根文件数 | 根目录数 | Dot-Dirs | 最大深度 | 备注 |
|------|-----|---------|---------|----------|---------|------|
| 2025-01 | 35 | 92 | 42 | 15 | 8 | 初始状态 |
| 2025-06 | 62 | 30 | 18 | 18 | 6 | 第一轮整理 |
| 2025-11 | 78 | 12 | 13 | 20 | 5 | 第二轮整理 |
| 2026-01 | 45 | 49 | 84 | 23 | 7 | 回退（教训） |
```

> Dot-Dirs 数量增长是正常的（新工具引入），不应视为负面指标。

## 10. 反模式篇：常见陷阱与实战教训

### 10.1 十二大反模式

| # | 反模式 | 症状 | 正确做法 |
|---|--------|------|----------|
| 1 | 根目录垃圾场 | 根目录 50+ 文件 | 按类型分流到子目录 |
| 2 | 过度嵌套 | src/main/java/com/... 7+ 层 | 控制在 4-6 层 |
| 3 | 按人分目录 | alice/, bob/ | 按功能/领域分 |
| 4 | 日期目录泛滥 | 2025-01/, 2025-02/... | 用 git 管理历史 |
| 5 | 大爆炸重组 | 一次性移动 500+ 文件 | 渐进式迁移 |
| 6 | 只移不修 | 移了文件不修引用 | 移动 + 修复 = 原子操作 |
| 7 | 备份目录堆积 | bak/, old/, backup/ | 用 git branch/tag |
| 8 | 空目录占位 | 提前创建未来可能用的目录 | 按需创建 |
| 9 | README 缺失 | 目录用途靠猜 | 每个主目录一个 README |
| 10 | 忽视 .gitignore | 缓存/构建产物入库 | 及时更新 .gitignore |
| 11 | **整理 dot-directories** | 移动 .claude/ 导致 AI 工具失效 | dot-dirs 是禁区，绝不整理 |
| 12 | **清理 dot-files** | 删除 .env 导致环境崩溃 | dot-files 需逐个评估，默认保留 |

### 10.2 来自真实项目的六条教训

**教训一：整理是一场持久战，不是一次战役**
> 本项目经历了 134→64→13→84 的目录数波动。每次"大整理"后，如果没有防御机制，3-6 个月就会回退。

**教训二：dot-directories 是工具链的命脉**
> 项目中有 23 个 dot-directories，涵盖 AI CLI（.claude, .amazonq, .cursor, .gemini, .opencode 等）、IDE（.vscode）、VCS（.git, .github）、缓存（.mypy_cache, .pytest_cache）。移动任何一个都可能导致对应工具完全失效，且排查成本极高。

**教训三：bak/ 目录是定时炸弹**
> 一个 bak/ 目录可能包含完整的 node_modules 副本（3800+ 文件/目录），占用数十 MB。用 git tag 替代文件系统备份。

**教训四：文档整理和代码整理要分开**
> 文档（.md）和代码（.py/.ts）的组织逻辑不同。文档按主题聚类，代码按功能/领域聚类。混在一起整理容易顾此失彼。

**教训五：自动化是唯一可持续的方案**
> 人工整理依赖纪律，纪律会衰减。只有 git hook + CI + 定期审计的三道防线才能真正持久。

**教训六：统计 dot-directories 时要区分对待**
> 评估根目录整洁度时，dot-directories 不应计入"混乱度"指标。它们的数量增长反映的是工具链丰富度，不是组织混乱。

## 11. 附录

### 11.1 常用命令速查

```bash
# ── 诊断类（所有命令均排除 dot-dirs）──

# 根目录非 dot 文件数
find . -maxdepth 1 -type f ! -name '.*' | wc -l

# 根目录非 dot 目录数
find . -maxdepth 1 -type d ! -name '.*' ! -name '.' | wc -l

# dot-directories 清单（仅查看，不操作）
find . -maxdepth 1 -type d -name '.*' ! -name '.' ! -name '.git' | sort

# 各目录文件数排名（排除 .git/node_modules/dot-dirs）
find . -not -path './.git/*' -not -path '*/node_modules/*' \
  -not -path './.*/*' -type f \
  | sed 's|/[^/]*$||' | sort | uniq -c | sort -rn | head -20

# 文件类型分布
find . -type f -not -path './.git/*' -not -path '*/node_modules/*' \
  | grep -oE '\.[^.]+$' | sort | uniq -c | sort -rn | head -20

# 最大嵌套深度
find . -type f -not -path './.git/*' -not -path '*/node_modules/*' \
  | awk -F/ '{print NF-1}' | sort -rn | head -1

# 大文件检测
find . -type f -size +5M -not -path './.git/*' | sort -k5 -rn

# 空目录检测
find . -type d -empty -not -path './.git/*'

# 重复文件名检测
find . -type f -not -path './.git/*' | xargs -I{} basename {} \
  | sort | uniq -cd | sort -rn | head -20

# ── 操作类 ──

# 安全移动（保留 git 历史）
git mv old/path new/path

# 批量移动某类型文件
find . -maxdepth 1 -name '*.py' ! -name 'conftest.py' \
  -exec git mv {} scripts/dev/ \;

# 清理临时文件（排除 dot-dirs）
find . -maxdepth 1 -type f \( -name '*.log' -o -name '*.tmp' -o -name '*.bak' \) \
  ! -name '.*' -delete
```

### 11.2 命名规范速查

| 类型 | 规范 | 示例 |
|------|------|------|
| 目录 | 小写 kebab-case 或 snake_case | `user-auth/`, `data_models/` |
| Python 文件 | snake_case | `data_processor.py` |
| JS/TS 文件 | camelCase 或 kebab-case | `dataProcessor.ts` |
| 组件文件 | PascalCase | `UserProfile.vue` |
| 配置文件 | 工具约定优先 | `tsconfig.json`, `.eslintrc` |
| 文档 | 大写或 kebab-case | `README.md`, `api-guide.md` |
| Dot-files | 工具约定，不可更改 | `.gitignore`, `.env` |
| Dot-dirs | 工具约定，不可更改 | `.vscode/`, `.github/` |

### 11.3 决策流程图

```
新文件/目录需要放置？
│
├─ 是 dot-file/dot-dir？
│  ├─ 是 → 🚫 不动。放在工具约定的位置（通常是项目根目录）
│  └─ 否 ↓
│
├─ 是配置文件？(.yaml/.toml/.ini/.json 且非数据)
│  ├─ 是 → config/
│  └─ 否 ↓
│
├─ 是文档？(.md/.rst/.txt 且非 README)
│  ├─ 是 → docs/ 下按主题分类
│  └─ 否 ↓
│
├─ 是脚本？(.py/.sh/.js 且为工具/自动化)
│  ├─ 是 → scripts/ 下按用途分类
│  └─ 否 ↓
│
├─ 是源代码？(业务逻辑)
│  ├─ 是 → src/ 下按功能/领域分类
│  └─ 否 ↓
│
├─ 是测试？
│  ├─ 是 → tests/ 下镜像源码结构
│  └─ 否 ↓
│
├─ 是生成产物？(报告/覆盖率/日志)
│  ├─ 是 → reports/ 或 var/log/，考虑 .gitignore
│  └─ 否 ↓
│
├─ 是临时文件？
│  ├─ 是 → temp/ 或 .gitignore，定期清理
│  └─ 否 ↓
│
└─ 无法分类 → 先放 misc/，下次审计时归类
```

## 结语

目录整理不是一次性的技术任务，而是一种持续的工程纪律。

核心要记住三件事：

1. **Dot-directories 是禁区** — 它们是工具链的命脉，数量增长是正常的，永远不要整理它们
2. **渐进优于激进** — 小步快跑，每步验证，随时可回退
3. **自动化是唯一可持续的方案** — 规范会被遗忘，习惯会衰减，只有机器不会偷懒

> 好的目录结构不会让你注意到它的存在，但坏的目录结构会让你每天都在骂人。
