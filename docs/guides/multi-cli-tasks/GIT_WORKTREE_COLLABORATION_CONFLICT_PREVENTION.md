## 使用说明

本规范用于预防并处理以下常见冲突：
- 配置文件冲突（如 pre-commit、lint、CI 配置）
- 同文件并发修改冲突
- 主 CLI 未同步已完成 worktree 导致重复劳动
- 任务文档放错位置导致冲突

落地时替换：`[项目名]`、`[主仓库路径]`、`CLI-X`。

## 文档目的

通过“规则前置 + 职责隔离 + 证据留痕”实现：
- 冲突少发生
- 冲突易定位
- 冲突可快速恢复

---

## 冲突治理总原则

1. **明确所有权**：每个目录/文件有 owner。
2. **职责分离**：Worker CLI 只改分配范围。
3. **集中配置管理**：全局配置由主 CLI 统一维护。
4. **任务文档分离**：任务定义与进度报告分离。
5. **先审后合**：所有变更先入 `dev`，再入 `main`。

---

## 分支策略（所有 CLI 必须遵循）

### 1. 分支基础规则
- **基准分支**: `dev`
- **生产分支**: `main`（仅从 `dev` 合并）
- **临时分支**: `feat/[模块名]-[cli标识]` 或 `fix/[模块名]-[cli标识]`

### 2. 工作区创建（Git Worktree）
```bash
# 主CLI初始化（仅首次执行）
git worktree add ./ai-workspace -b dev origin/dev

# 子CLI创建专属工作区
git worktree add ./worker-payment -b feat/payment-cli-x dev
```

---

## PR 管理规则

### 1. PR 创建（Worker CLI 执行）
```bash
gh pr create --base dev --head [分支名] \
  --title "[type(scope)]: 描述" \
  --body "AI CLI: [CLI名称] | 生成模块: [模块名]"
```

### 2. PR 审核与合并（主 CLI 执行）
- 主 CLI 审核后合并到 `dev`
- `dev` 至少累计 2 个有效 PR 后，方可合并至 `main`

---

## 提交信息规范（AI 生成必须遵循）

- 格式：`type(scope): short description`（英文）
- type 建议：`feat`/`fix`/`docs`/`refactor`/`chore`
- scope 必须与任务分配模块一致

## 问题1：配置文件冲突

### 问题描述
多个 CLI 修改全局配置（如 `.pre-commit-config.yaml`）导致合并冲突。

### 标准解法
- 主 CLI 独占维护全局配置
- Worker CLI 不修改全局配置，必要时通过局部参数绕过

---

## 问题2：任务分配冲突

### 问题描述
多个 Worker CLI 修改同一文件导致 `Both modified`。

### 标准解法
1. 建立 `.FILE_OWNERSHIP` 或同等映射
2. 目录级隔离：按模块分配
3. 跨 owner 修改必须走申请与审批

---

## 问题3：主CLI未同步已完成worktree

### 问题描述
主 CLI 未及时同步已完成分支，导致基线落后与重复开发。

### 标准解法
主 CLI 开工前强制 Pre-flight：
```bash
git worktree list
git fetch --all
git log HEAD..origin/dev --oneline
```
发现已完成分支时，先完成审查合并再进入新任务。

---

## 问题4：任务文档方式冲突

### 问题描述
使用共享 README 记录任务与进度，造成高频冲突。

### 标准解法
- `TASK.md`：任务说明（主 CLI 下发）
- `TASK-REPORT.md`：进度报告（Worker CLI 更新）
- `TASK-*-REPORT.md`：阶段/最终报告
- 项目 README 保持项目级内容，不记录 Worker 任务细节

---

## 问题5：依赖同步冲突

### 问题描述
多 worktree 环境下依赖版本不一致，导致构建与测试结果分裂。

### 标准解法
- 统一锁文件策略
- 合并前强制依赖一致性检查
- 必要时使用集中依赖缓存/镜像，避免“每个工作区一个版本”

---

## 违规处理

### 1. Worker CLI 提交 PR 到 main 分支
- **处理**: 主 CLI 直接关闭 PR，要求重新基于 `dev` 提交

### 2. 提交信息不符合规范
- **处理**: 主 CLI 驳回 PR，要求修正后重新提交

### 3. AI 生成代码未验证
- **处理**: 主 CLI 要求执行验证命令（如 `tsc`/`pytest`）并附验证结果

---

## 执行优先级

### 立即执行
1. 明确分支策略（`dev` 唯一入口）
2. 统一 PR 模板与提交规范
3. 建立任务文档双文件机制（TASK + REPORT）
4. 建立 owner 映射并执行冲突扫描

### 持续执行
1. 每日 Pre-flight
2. 每轮合并后复盘并更新规范
