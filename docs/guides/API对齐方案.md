一、先让 AI 完成「标准化资产梳理」（对齐的基础：统一前后端 “语言”）
核心目标：让 AI 先把前后端的 “数据口径、接口约定、业务逻辑” 梳理成统一的标准化文档，避免 “各说各话”。
1. 指挥 AI 梳理后端 API 资产（量化系统重点关注）
指令方向：“解析后端所有API 的 mock 函数/数据、接口定义（请求方式 / 路径 / 参数），按本量化系统的菜单/页面功能（如行情模块、交易模块、风控模块、账户模块、报表模块）分类，生成一份《量化系统 API 标准化文档》，要求包含：① 每个 API 的业务含义（比如 “获取沪深 300 分钟级行情”“提交限价单”）；② 请求参数的必填 / 可选、数据类型（重点标注量化特有的精度要求，如浮点数保留位数、金额单位）；③ mock 响应数据的字段结构；④ 预期的错误码及含义（如风控拦截码、接口超时码）；⑤ API 的访问权限（如仅管理员 / 普通用户可调用）。”
关键要求：让 AI 识别 API 中的重复 / 冗余接口（比如同个行情数据有多个接口）、格式不统一的字段（比如 “tradeTime” 和 “trade_time” 混用），并标注出来。
2. 指挥 AI 梳理前端页面资产
指令方向：“梳理前端所有页面的功能模块（按业务域和页面层级），生成《前端页面 - 功能 - 数据诉求映射表》，要求包含：① 每个页面的核心功能（如 “行情展示”“下单操作”“风控规则配置”）；② 每个功能需要的数据源（比如 “行情展示需要 K 线数据、最新成交价、成交量”）；③ 功能对应的交互行为（如 “点击下单按钮需要提交数据、等待返回、刷新账户持仓”）；④ 前端预期的数据格式（比如 K 线数据的时间戳格式、金额字段的展示精度）。”
3. 指挥 AI 完成「页面 - API」精准映射
指令方向：“基于上述两份文档，生成《量化系统页面 - API 映射总表》，明确：① 前端每个页面的每个功能按钮 / 数据展示区域，对应后端哪一个 / 多个 API；② 标注映射不匹配的地方（比如前端需要 “持仓盈亏计算” 数据，但后端无对应 API；后端有 “历史委托单查询” API，但前端无对应页面功能）；③ 标注数据格式冲突点（比如前端预期 “金额” 字段是 Number 类型，后端 mock 返回 String 类型）。”

二、让 AI 按「分阶段对齐策略」执行（先核心后非核心，避免全量混乱）
量化系统的核心是 “交易、行情、风控”，优先让 AI 聚焦核心模块对齐，而非一次性搞定 200+API，降低复杂度。
1. 第一阶段：核心模块端到端对齐（行情 / 交易 / 风控）
指令方向：“聚焦行情展示、限价 / 市价下单、风控规则校验这三个核心模块，优先处理这三类功能对应的 API 与前端页面的对齐：① 先校验后端核心 API 的 mock 数据格式，是否完全匹配前端核心页面的展示 / 交互诉求；② 生成《核心模块数据对齐整改清单》，明确需要调整的后端 mock 数据格式、前端数据解析逻辑；③ 给出对齐优先级（比如 “下单接口的参数校验规则” 优先级高于 “历史行情导出接口”）。”
关键：让 AI 识别量化场景的特殊对齐点（如行情数据的实时性要求、下单接口的参数校验规则、风控阈值的数值范围）。
2. 第二阶段：非核心模块批量对齐（报表 / 账户 / 系统设置）
指令方向：“针对报表统计、账户信息、系统设置等非核心模块，按 “数据格式统一化” 原则批量处理：① 提取所有非核心 API 的通用数据字段（如时间戳、用户 ID、金额），生成统一的字段命名 / 格式规范；② 识别前端非核心页面的通用数据解析逻辑，生成批量适配方案；③ 标注无需对齐的边缘 API（如仅测试用的 mock 接口），暂时屏蔽。”
3. 第三阶段：异常场景对齐（量化系统高可靠性要求）
指令方向：“梳理量化系统所有可能的异常场景（API 超时、数据返回为空、风控拦截、权限不足、行情数据延迟），生成《异常场景前后端对齐方案》：① 明确后端每个异常对应的错误码 / 提示文案；② 明确前端对应的错误展示方式（如弹窗、页面提示、日志记录）；③ 明确异常恢复逻辑（如 API 超时后自动重试次数、风控拦截后是否允许重新提交）。”

三、让 AI 辅助「验证与闭环」（确保功能真的能用）
1. 指挥 AI 生成验证方案（量化场景针对性）
指令方向：“基于核心模块的对齐结果，生成《量化系统功能验证清单》，要求：① 按 “页面 - 功能 - API” 维度，列出每个功能的验证点（比如 “下单功能需验证：参数合法时返回成功、参数超限触发风控拦截、无权限时返回错误码”）；② 针对量化特有的边界场景补充验证点（如大额下单、高频行情请求、跨交易日数据查询）；③ 明确验证方式（手动操作验证、mock 数据替换验证）。”
2. 指挥 AI 识别对齐中的重复问题，沉淀规范
指令方向：“汇总对齐过程中出现的所有问题（如字段名不统一、API 命名无规则、错误码混乱），生成《量化系统前后端统一规范》，包含：① 接口命名规范（如 “行情接口统一前缀 /api/market/”）；② 数据字段字典（如 “金额字段统一命名为 amount，类型为 Number，保留 4 位小数”）；③ 错误码体系（如风控类错误码以 100 开头、系统类以 200 开头）；④ 后续新增 API / 页面必须遵循的规则。”
3. 指挥 AI 辅助自动化校验（减少人工重复工作）
指令方向：“基于统一规范，设计一套 “前后端数据对齐自动化校验逻辑”（无需编写具体代码，只需明确校验规则）：① 校验维度（字段名一致性、数据类型一致性、必填参数完整性、错误码匹配性）；② 触发时机（后端 API 更新后、前端页面迭代后）；③ 校验结果输出形式（问题清单、通过率报表）。”
四、让 AI 主导「迭代优化」（持续收敛问题）
1. 指令方向：“每次验证后，汇总未对齐的问题，生成《问题收敛计划表》：① 按 “严重程度（阻断功能 / 不影响核心功能）” 分类；② 明确每个问题的整改责任（前端调整 / 后端调整）；③ 给出整改建议（比如 “前端将 String 类型的金额转为 Number 类型”“后端统一时间戳格式为毫秒级”）。”
2. 指令方向：“针对量化系统的性能诉求（如高频行情接口的响应速度），让 AI 分析现有 API 的 mock 数据返回逻辑，给出《性能优化建议》（如行情数据分页返回、高频接口增加缓存逻辑、前端数据懒加载）。”

核心指挥 AI 的原则总结
先标准化，再对齐：不让 AI 直接改代码，先让它把前后端的 “资产、约定、规则” 梳理清楚，这是对齐的前提；
先核心，再边缘：量化系统核心功能（交易 / 行情 / 风控）是生命线，优先让 AI 聚焦这些模块，避免全量铺开导致混乱；
聚焦 “约定” 而非 “代码”：指挥 AI 生成统一的规范、映射表、验证清单，而非直接写代码，让前后端开发按 AI 输出的 “约定” 调整，确保对齐；
量化场景特殊化：所有指令中必须融入量化系统的特性（数据精度、实时性、风控、异常处理），避免通用 web 对齐方案不适用。
按这个思路指挥 AI，AI 会先帮你理清 “对齐什么、怎么对齐、验证什么”，再逐步落地，最终实现前后端数据闭环、功能正常使用。
